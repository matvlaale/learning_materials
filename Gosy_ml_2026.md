## 1 билет: Архитектурные особенности современных микропроцессоров. Особенности организации подсистемы памяти и динамической памяти, их влияние на производительность суперкомпьютеров при решении различных задач, в том числе задач обработки больших графов.

### 1. Общие понятия архитектуры микропроцессоров

**Микропроцессор (процессор)** – главный компонент компьютера, обрабатывающий информацию, выполняющий программы и управляющий другими устройствами системы.

**Ядро** – основа микропроцессора, состоящая из миллионов транзисторов на кристалле кремния. Микропроцессор разбит на специальные ячейки – **регистры общего назначения (РОН)**.

**Кэш-память** – внутренняя память процессора, используемая как буфер для ускорения доступа к часто используемым данным и инструкциям.

Современные микропроцессоры характеризуются следующими особенностями:
- один процессор, как правило, содержит несколько ядер;
- используется многоуровневая система кеширования доступа к памяти:
    - L1 кэш – очень быстрый, для одного ядра;
    - L2 кэш – больше, но медленнее, также для одного ядра;
    - L3 кэш – ещё больше, общий для всего процессора;
- кешируются как данные, так и инструкции;
- процессор может переупорядочивать или смешивать инструкции и доступы к памяти;
- появляются физические акселераторы для некоторых вычислительных задач (например, для криптографии или архивирования/кодирования);
- инструкции конвейеризируются: разбиваются на мелкие части (получение инструкции, декодирование, чтение операнда из памяти, вычисление выражения, запись результата). У современных процессоров количество таких стадий составляет 16–32, что ускоряет выполнение инструкций;
- появляются специализированные процессоры, например GPU, имеющие огромное число ядер (которые одновременно исполняют одну и ту же инструкцию), специальные правила доступа к памяти и большое количество регистров.

### 2. Типы архитектур процессоров

#### 2.1. CISC (Complex Instruction Set Computer)
- **Особенности**: нефиксированная длина команд, кодирование арифметических действий в одной команде, небольшое число регистров, многие из которых выполняют строго определённую функцию.
- **Примеры**: x86 (IA-32), x86_64 (AMD64).
- **Плюсы**: высокая производительность благодаря тому, что несколько команд могут быть заменены одной аналогичной.
- **Минусы**: большая цена по сравнению с RISC из‑за более сложной архитектуры, команды сложнее раскодировать.

#### 2.2. RISC (Reduced Instruction Set Computer)
- **Особенности**: упрощённые инструкции для увеличения быстродействия; первые RISC‑процессоры не имели даже инструкций умножения/деления и не поддерживали работу с числами с плавающей запятой.
- **Плюсы**:
    - константная длина команды, меньшее количество схожих инструкций ⇒ уменьшение цены и энергопотребления;
    - большее число регистров;
    - скорость выше по сравнению с CISC (выполнение до 10 раз быстрее).
- **Минусы**: совместимость. x86_64 остаётся лидером в desktop‑сегменте по историческим причинам (старые программы и игры работают только на x86, поэтому новые системы должны сохранять совместимость).

#### 2.3. MISC (Minimal Instruction Set Computer)
- **Особенности**: ещё более простая архитектура, предназначенная для уменьшения цены и энергопотребления. Используется в IoT‑сегменте и недорогих компьютерах (например, роутерах).
- **Плюсы**:
    - спекулятивное исполнение команд (выполнение команды до того, как станет известно, понадобится ли она) для увеличения производительности;
    - наилучшая эффективность по сравнению с CISC и RISC;
    - простота.
- **Минусы**: сложность написания программ под различные процессоры.

#### 2.4. VLIW (Very Long Instruction Word)
- **Особенности**: архитектура с несколькими вычислительными устройствами; одна инструкция содержит несколько операций, которые должны выполняться параллельно. Является разновидностью CISC со спекулятивным исполнением, но спекуляция выполняется во время компиляции, а не во время работы программы.
- **Преимущество в безопасности**: для VLIW‑процессоров невозможны уязвимости Meltdown и Spectre, связанные с ошибками реализации спекулятивного выполнения на этапе работы.
- **Примеры**: Intel Itanium, Эльбрус‑3.
- **Плюсы**:
    - сильно упрощается архитектура процессора – задача распределения вычислительных устройств перекладывается на компилятор;
    - отсутствие больших и сложных узлов ⇒ сильно снижается энергопотребление.
- **Минусы**:
    - код обладает невысокой плотностью;
    - сложность программирования (компиляторы сильно привязаны к конкретным процессорам).

#### 2.5. Другая классификация (по принципу «поток команд – поток данных»)
1. **Скалярные (SISD – Single Instruction Single Data)** – обрабатывают один элемент данных за одну инструкцию.
2. **Векторные (SIMD – Single Instruction, Multiple Data)** – обрабатывают несколько данных, выполняя над ними одну операцию.
3. **MISD (Multiple Instructions, Single Data)** – несколько вычислительных устройств выполняют различные операции над одним данным.
4. **MIMD (Multiple Instructions, Multiple Data)** – несколько вычислительных устройств выполняют различные операции над различными данными (например, многопроцессорная система из скалярных процессоров).
5. **Конвейер** – над данными, движущимися через конвейер, выполняются различные операции на каждом шаге.

### 3. Подсистема памяти: иерархия и организация

**Подсистема памяти** – совокупность устройств для запоминания, хранения и выдачи информации в виде цифровых кодов. Отдельные устройства называются запоминающими устройствами (ЗУ). Наиболее распространён иерархический принцип построения.

Уровни иерархии памяти (от более высокого к низкому):
1. **Регистры** – самые быстрые, малой ёмкости, высокая стоимость на бит.
2. **Кэш-память** (SRAM).
3. **Основная память (оперативная)** – DRAM.
4. **Твердотельные и магнитные диски**.
5. **Оптические диски, ЗУ на магнитных лентах**.

Три верхних уровня образуют внутреннюю память МПС, нижние – внешнюю (вторичную) память.

**Свойства иерархии** (по мере движения вниз):
- уменьшается соотношение стоимость/бит;
- возрастает ёмкость;
- растёт время доступа;
- уменьшается частота обращения к памяти со стороны процессора.

### 4. Организация кэш-памяти

- Доступ в кэш значительно быстрее, чем в оперативную память.
- Загрузка данных из памяти в кэш происходит **кэш-линиями** (блоками фиксированного размера).
- При выборе блока для замещения могут использоваться:
    - случайный выбор;
    - запись в последовательном порядке (циклический);
    - на место блока, который не адресовался дольше других (LRU).

### 5. Организация оперативной памяти (DRAM)

- **RAM** – Random Access Memory: доступ к любому адресу за примерно одинаковое время.
- **DRAM** – Dynamic RAM: динамическая, содержимое необходимо периодически освежать (регенерация), так как заряд в конденсаторе ячейки постепенно утекает.
- **SRAM** – Static RAM: энергозависима, но не требует постоянной регенерации; быстрее и дороже DRAM.

**Почему основная память – DRAM, а не SRAM?**
В ячейке DRAM используется в несколько раз меньше элементов, поэтому такая память кратно дешевле и имеет более высокую плотность (больше ячеек на кристалле). Недостаток – необходимость регенерации (занимает до 10% пропускной способности) и большее время доступа.

- **SDRAM** – Synchronous DRAM: работает синхронно под управлением тактового сигнала контроллера, доступ потактовый.
- **DDR** – Double Data Rate: удвоенная скорость передачи данных относительно частоты тактового сигнала (шины памяти).

**Устройство модуля памяти:**
Модуль памяти состоит из набора чипов. Чипы состоят из **банков**. Банк представляет собой таблицу (матрицу), в каждой ячейке которой хранится бит данных. Для доступа к биту указывается адрес: номер чипа, номер банка, номер строки и номер столбца.

**Управляющие линии и сигналы:**
- **RAS** (Row Address Strobe) – сигнал о том, что на адресные линии поступит номер строки.
- **CAS** (Column Address Strobe) – сигнал о передаче адреса столбца.
- **Precharge** (предзаряд) – операция, необходимая для закрытия текущей строки и открытия другой; занимает время, сопоставимое с чтением адреса строки и столбца (порядка 10–15 нс).

**Латентность доступа** складывается из задержек RAS, CAS и предзаряда. Чтение данных из разных строк занимает больше времени, чем из одной строки, так как требуется передавать адрес строки и выполнять предзаряд.

**Методы повышения пропускной способности:**
- **Пакетный режим (burst)**: один раз передаётся адрес столбца, и считывается подряд сразу пакет из нескольких столбцов.
- **Чередование (interleaving)**: идущие подряд данные распределяются по разным банкам (адрес X находится в банке `X % k`, где k – число банков). Все банки работают параллельно, что повышает пропускную способность при последовательной выборке.
- **Вывод**: читать данные из памяти эффективнее подряд (последовательно).

### 6. Ключевые характеристики производительности памяти

- **Bandwidth (пропускная способность)** – максимальная скорость передачи данных, измеряется в байтах в секунду.
- **Latency (латентность доступа)** – задержка от момента инициирования запроса до начала поступления данных. Для оперативной памяти составляет десятки наносекунд.

Время на чтение N байт данных:  
`T = L + N / BW`, где `L` – латентность, `N/BW` – время на передачу данных после начала передачи.  
- Если N маленькое, доминирует латентность.
- Если N большое, доминирует пропускная способность.

### 7. Оптимизации производительности (кратко)

- Развёртка циклов (loop unrolling).
- Конвейеризация.
- Использование по максимуму пропускной способности памяти.
- Организация эффективного доступа к данным (последовательный, с учётом банков, пакетный режим).

### 8. Влияние архитектуры памяти на производительность при решении задач обработки больших графов

Для графовых алгоритмов характерны следующие проблемы:

1. **Data-driven computations** – зависимость вычислений от данных (топологии графа). Невозможность применения методов статического распараллеливания вычислений.
2. **Poor locality** – низкая пространственно-временная локализация обращений к памяти. Обращения разбросаны по всему адресному пространству, что приводит к частым промахам кэша и обращениям к медленной DRAM.
3. **High data access to computation ratio** – преобладание команд доступа к памяти над командами выполнения арифметических операций. Соотношение производительность/пропускная способность памяти постоянно растёт (вычислительные возможности увеличиваются быстрее, чем скорость памяти), поэтому графовые задачи становятся всё более «узким местом» по памяти.

**Сравнение типов нагрузок:**
- **HPL** (High Performance Linpack, используемый в рейтинге Top500) – вычислительно интенсивная задача. Сложность `O(n^3)`, количество доступов к памяти `O(n^2)`. Производительность достигает десятков процентов от пиковой.
- **HPCG** (High Performance Conjugate Gradients) – более приближена к реальным приложениям, в том числе графовым. Сложность `O(n)`, количество доступов к памяти `O(n)`. Производительность составляет лишь единицы процентов от пиковой, что демонстрирует сильную зависимость от подсистемы памяти.

Таким образом, для эффективного решения задач обработки больших графов на суперкомпьютерах критически важны низкая латентность и высокая пропускная способность памяти, а также техники, улучшающие локальность (блочное разбиение, переупорядочивание вершин, использование кэша). Архитектурные особенности современных процессоров (многоядерность, иерархический кэш, спекулятивное выполнение) не всегда могут компенсировать нерегулярный доступ к памяти, характерный для графов.

## 2 билет: Высокоскоростные коммуникационные сети. Основные характеристики: задержка, пропускная способность, темп выдачи сообщений, бисекционная пропускная способность. Технология RDMA. Топологии сетей: тор, жирное дерево, Flattened Butterfly, Dragonfly.

### 1. Основные характеристики коммуникационных сетей

- **Задержка передачи сообщений (latency)** – время от момента отправки сообщения до его полного получения адресатом (включает стартовые задержки, задержки на хопах и время передачи данных).
- **Пропускная способность (bandwidth)** – максимальная скорость передачи данных, измеряется в байтах/с.
- **Темп выдачи сообщений (message rate)** – количество сообщений, которое сеть может передать в единицу времени. Связь с пропускной способностью: `MessageRate = BW(L) / L`, где `L` – размер сообщений. На практике `MessageRate < L / BW`.
- **Бисекционная пропускная способность (bisection bandwidth)** – пропускная способность каналов связи через **бисекционную плоскость** (минимальный разрез, который делит сеть на две равные связные части). В случае равномерных случайных посылок all‑to‑all каждый узел посылает сообщение через бисекционную плоскость с вероятностью ½. Для линейной масштабируемости требуется `N/2` линков в бисекционной плоскости.
- **Топология** – способ соединения узлов и коммутаторов.
- **RDMA** – технология удалённого прямого доступа к памяти.

**Модель времени передачи сообщения:**  
`t_comm = t_start + d * t_hop + size * t_bw`, где  
- `t_start` – стартовая задержка (накладные расходы на инициализацию),  
- `d` – количество хопов (промежуточных узлов),  
- `t_hop` – задержка на один хоп,  
- `size` – размер сообщения,  
- `t_bw` – время передачи единицы данных (обратная пропускной способности).

**Эффективность программ достигается за счёт:**
- агрегации больших сообщений (уменьшение количества отправок),
- минимизации расстояний пересылок (меньше хопов),
- минимизации объёмов коммуникаций.

### 2. Технология RDMA (Remote Direct Memory Access)

RDMA позволяет одному узлу выполнять чтение или запись в память другого узла **без участия CPU** удалённой стороны и без копирования данных в буферы ОС. Данные передаются напрямую между сетевым адаптером и памятью приложения.

**Типы операций RDMA:**
- **Удалённая запись** (remote write) – отправка данных в память удалённого узла.
- **Удалённое чтение** (remote read) – чтение данных из памяти удалённого узла.

**Преимущества:** снижение задержки, уменьшение нагрузки на CPU, высокая пропускная способность. RDMA широко применяется в высокоскоростных сетях (InfiniBand, RoCE, iWARP).

### 3. Топологии сетей

#### 3.1. Полносвязная топология (fully connected)
- Максимальная задержка `Dmax = 1`.
- Идеальная топология: нет конкуренции, наименьшая задержка.
- Высокая цена: `O(N)` портов на узел, `O(N²)` линков. Не масштабируется.

#### 3.2. Тор (kD‑тор, k‑dimensional torus)
- Определяется как произведение одномерных решёток: размерности `n1 × n2 × … × nk`.
- Максимальная задержка: `Dmax = [n1/2] + [n2/2] + … + [nk/2]`.
- Масштабируется, большое количество вариантов путей.
- Бисекционная ширина: `2N / N_max` (зависит от размеров).
- Относится к low‑radix топологиям (коммутаторы с небольшим числом портов).
- **Примеры использования:** суперкомпьютеры «Ломоносов» (InfiniBand 4xQDR), Tofu (6D‑тор).

#### 3.3. Жирное дерево (FatTree)
- Непрямая топология (используются коммутаторы, узлы не соединены напрямую).
- Другое название – Folded Clos.
- Параметры: `k` – количество портов коммутатора, `c` – количество узлов, подключаемых к одному коммутатору.
- `Dmax = 4`.
- Масштабируется, большое количество вариантов путей.
- Бисекционная ширина: `N/2` (или `N/4` в зависимости от конфигурации).
- Бывает неблокирующая и блокирующая (при перепроектировании).
- **Пример:** сеть InfiniBand в суперкомпьютере «Ломоносов».

#### 3.4. Flattened Butterfly (сплюснутая бабочка)
- Предложена в 2007 году. Обозначение: `FB(c, k‑ary, n‑flat)`, где:
    - `c` – количество узлов, подключаемых к одному коммутатору;
    - `k` – размерность (radix) коммутатора;
    - `n` – количество измерений (dimensions).
- Максимальная задержка: `Dmax = 2 + n - 1` (на один хоп меньше, чем в обычной бабочке за счёт «сплющивания»).
- Каждое измерение внутри полностью связано (full‑mesh).
- Позволяет уменьшить количество коммутаторов и длину путей по сравнению с обычной бабочкой.
- **Пример:** 8‑ary 2‑flat (c=8, 8‑арная, 2‑плоскостная). Рисунок в материалах показывает, что flattened butterfly получается из обычной бабочки удалением промежуточных коммутаторов и прямыми связями.

#### 3.5. Dragonfly (Стрекоза)
- Предложена в 2008 году, иерархическая топология.
- **Иерархия:**
    - Внутри группы узлов – топология «все со всеми» (полносвязная).
    - Группы между собой также соединены по принципу «все со всеми».
- Параметры:
    - `p` – количество узлов на один роутер,
    - `a` – количество роутеров в группе,
    - `h` – количество линков роутера для соединения с другими группами.
- `Dmax = 3`.
- Адаптивные маршруты (возможность обхода перегруженных каналов).
- Линейно растущая бисекционная ширина (с ростом числа узлов).
- **Применение:** многие современные суперкомпьютеры в Top500 (например, система, занявшая 5‑е место в июне 2021 года, – 89.8 Пфлопс). Коммутаторы 64‑radix Dragonfly позволяют строить группы по 512 узлов.
- **Сравнение стоимости** (приведено в материале): Dragonfly даёт лучшее соотношение производительности/стоимости по сравнению с тором и жирным деревом при большом количестве узлов.

## 3 билет: Образцы (паттерны) проектирования, их классификация и способ описания. Пример образца проектирования.

### 1. Определение и назначение паттернов проектирования

**Паттерн проектирования (design pattern)** – многократно применяемая архитектурная конструкция, предоставляющая проверенное решение общей проблемы проектирования в рамках конкретного контекста. Паттерн не является готовым кодом, а представляет собой общую концепцию, шаблон или «микроархитектуру», описывающую взаимодействие объектов и классов.

**Цели использования паттернов:**
- **Повторное использование решений** – не «изобретать велосипед», использовать опыт и лучшие практики.
- **Общий словарь** – единая терминология для разработчиков (фраза «здесь нужно использовать "Адаптер"» сразу проясняет суть решения).
- **Повышение гибкости и удобства сопровождения** – система легче поддаётся модификации и расширению.
- **Стандартизация и качество** – формализованные, проверенные временем решения повышают надёжность архитектуры.

Концепция паттернов в программировании была вдохновлена работами архитектора Кристофера Александера и популяризирована «бандой четырёх» (Gang of Four, GoF) – Эрихом Гаммой, Ричардом Хелмом, Ральфом Джонсоном и Джоном Влиссидесом в книге «Design Patterns: Elements of Reusable Object-Oriented Software» (1994).

### 2. Основные элементы описания паттерна (по GoF)

1. **Имя (Pattern Name)** – уникальное, осмысленное название, позволяющее общаться на высоком уровне абстракции.
2. **Задача (Intent/Problem)** – описание проблемы и контекста применения.
3. **Решение (Solution)** – абстрактное описание структуры паттерна (классы, объекты, их отношения и взаимодействия).
4. **Результаты и последствия (Consequences)** – анализ компромиссов, преимуществ и недостатков (влияние на производительность, гибкость, сложность, повторное использование).

**Канонический шаблон описания (полный):**
1. Название и классификация.
2. Назначение.
3. Также известен как.
4. Мотивация (сценарий).
5. Применимость.
6. Структура (UML).
7. Участники.
8. Отношения.
9. Результаты.
10. Реализация.
11. Пример кода.
12. Известные применения.
13. Родственные паттерны.

### 3. Классификация паттернов проектирования (GoF)

#### 3.1. Порождающие паттерны (Creational)
Абстрагируют процесс создания объектов, инкапсулируют знания о конкретных классах.
- **Singleton (Одиночка)** – гарантирует единственный экземпляр класса и глобальную точку доступа.
- **Factory Method (Фабричный метод)** – определяет интерфейс создания объекта, но позволяет подклассам решать, экземпляр какого класса создавать.
- **Abstract Factory (Абстрактная фабрика)** – интерфейс для создания семейств взаимосвязанных объектов.
- **Builder (Строитель)** – отделяет конструирование сложного объекта от его представления.
- **Prototype (Прототип)** – создание новых объектов клонированием прототипа.

#### 3.2. Структурные паттерны (Structural)
Описывают композицию классов и объектов для образования гибких структур.
- **Adapter (Адаптер)** – преобразует интерфейс одного класса в интерфейс другого.
- **Bridge (Мост)** – отделяет абстракцию от реализации, позволяя им изменяться независимо.
- **Composite (Компоновщик)** – группирует объекты в древовидные структуры «часть-целое».
- **Decorator (Декоратор)** – динамически добавляет обязанности объектам.
- **Facade (Фасад)** – предоставляет унифицированный интерфейс к подсистеме.
- **Flyweight (Приспособленец)** – поддерживает много мелких объектов через разделение.
- **Proxy (Заместитель)** – предоставляет суррогат для контроля доступа к объекту.

#### 3.3. Поведенческие паттерны (Behavioral)
Определяют алгоритмы и способы взаимодействия между объектами.
- **Observer (Наблюдатель)** – определяет зависимость «один‑ко‑многим»; при изменении состояния одного объекта все зависящие уведомляются и обновляются автоматически.
- **Strategy (Стратегия)** – инкапсулирует семейство взаимозаменяемых алгоритмов.
- **Command (Команда)** – инкапсулирует запрос как объект (параметризация, очереди, поддержка отмены).
- **Chain of Responsibility (Цепочка обязанностей)** – передача запроса по цепочке обработчиков.
- **Iterator (Итератор)** – последовательный доступ к элементам коллекции без раскрытия её представления.
- **State (Состояние)** – позволяет объекту изменять поведение при изменении внутреннего состояния.

### 4. Пример образца проектирования: Singleton (Одиночка)

#### Назначение
Гарантирует, что у класса есть единственный экземпляр, и предоставляет глобальную точку доступа к нему.

#### Задача
Обеспечить контроль над созданием объекта, когда наличие более одного экземпляра недопустимо или нелогично (например, доступ к файлу конфигурации, пулу соединений с БД, логгеру). Обычный конструктор всегда возвращает новый объект, что не позволяет решить задачу напрямую.

#### Решение (общая структура)
1. **Сокрытие конструктора** – сделать конструктор приватным (private), чтобы предотвратить создание экземпляров извне.
2. **Статический метод доступа** – публичный статический метод (например, `getInstance()`), отвечающий за жизненный цикл единственного экземпляра.
3. **Статическая переменная** – внутри класса хранит ссылку на единственный экземпляр. При первом вызове `getInstance()` создаётся объект и сохраняется; при последующих вызовах возвращается сохранённый.

#### Пример (псевдокод на Java-подобном языке)

Класс `Database`, представляющий подключение к базе данных:

```
public class Database {
    // 1. Приватное статическое поле для хранения единственного экземпляра
    private static Database instance;

    // 2. Приватный конструктор (запрет создания извне)
    private Database() {
        // Здесь код для подключения к базе данных
        System.out.println("Подключение к базе данных установлено.");
    }

    // 3. Публичный статический метод - глобальная точка доступа
    public static Database getInstance() {
        // Ленивая инициализация: экземпляр создаётся при первом вызове
        if (instance == null) {
            instance = new Database();
        }
        return instance;
    }

    // Бизнес-логика класса
    public void executeQuery(String sql) {
        System.out.println("Выполняется запрос: " + sql);
    }
}
```

**Клиентский код:**

```
public class Main {
    public static void main(String[] args) {
        // Невозможно: конструктор приватный
        // Database db1 = new Database();

        // Получение единственного экземпляра
        Database db1 = Database.getInstance();
        db1.executeQuery("SELECT * FROM users");

        // Попытка получить ещё один экземпляр
        Database db2 = Database.getInstance();
        db2.executeQuery("SELECT * FROM products");

        // Проверка: db1 и db2 указывают на один и тот же объект
        if (db1 == db2) {
            System.out.println("Это один и тот же экземпляр класса Database.");
        }
    }
}
```

#### Результаты и последствия

**Преимущества:**
- Контролируемый доступ к единственному экземпляру.
- Глобальная точка доступа, более безопасная, чем глобальная переменная.
- «Ленивая» инициализация (создание ресурсоёмкого объекта откладывается до момента реальной необходимости).

**Недостатки:**
- Усложнение тестирования (глобальное состояние связывает тесты).
- Проблемы в многопоточной среде (наивная реализация не потокобезопасна; требуются синхронизация или двойная проверка блокировки).
- Нарушение принципа единственной ответственности (паттерн решает две задачи: гарантия единственности и глобальный доступ).

## 4 билет: Проблемы и методы эффективной организации параллельных вычислений при суперкомпьютерном решении задач обработки больших графов.

### 1. Типичная проблема на примере простой операции с графом

Рассмотрим цикл по рёбрам графа, в котором для каждого ребра вычисляется некоторое значение `r`, которое затем добавляется к переменным в вершинах, инцидентных этому ребру.

```
for(int ie=0; ie<NE; ++ie) {
    double r = Calc(ie);
    X[E[ie].v[0]] += r;
    X[E[ie].v[1]] += r;
}
```

Здесь `E` – массив рёбер размера `NE`, `v` в структуре ребра – номера инцидентных вершин, `X` – массив значений переменной во всех вершинах.

**Проблема:** непосредственно распараллелить этот цикл (например, директивой `#pragma omp parallel for`) нельзя из‑за **data race** – одновременной записи нескольких потоков в одну и ту же ячейку `X`.

### 2. Методы решения проблемы для общей памяти (OpenMP)

- **Дублирование вычислений:** сделать цикл по вершинам и дважды вызывать `Calc` для каждого ребра (по одному разу на каждую инцидентную вершину).  
  *Минус:* в два раза больше вычислений.

- **Атомарные операции:** добавить `#pragma omp atomic` перед строками `X[...] += r`.  
  *Минусы:* накладные расходы на атомики, ограниченная применимость.

- **Отдельный массив:** сохранять результаты `r` во временный массив, затем в отдельном цикле обновить `X`.  
  *Минусы:* дополнительный большой массив – расход памяти, лишний memory traffic.

- **Раскраска графа:** разбить рёбра на цвета так, чтобы рёбра одного цвета не имели общих вершин. Внешний цикл по цветам, внутренний по рёбрам текущего цвета – внутренний цикл легко параллелится.  
  *Минус:* нелокальность данных (рёбра одного цвета разбросаны в памяти).

- **Декомпозиция (разбиение на блоки):** разбить вершины между потоками (или MPI‑процессами). Рёбра, соединяющие вершины из разных блоков, образуют интерфейс.
    - **Обработка интерфейса последовательно** – минус: деградация с ростом числа нитей.
    - **Дублирование вычислений для интерфейсных рёбер** – минус: в два раза больше вычислений по интерфейсу, деградация (в меньшей степени) с ростом числа нитей.
    - **Многоуровневая декомпозиция** – сложнее, но может дать лучший баланс.

### 3. Подходы для распределённой памяти (MPI)

При использовании MPI обязательно выполняется **декомпозиция расчётной области** (графа). Каждый процесс получает свою подобласть.

**Термины:**
- **Внутренние элементы** – свои, не имеющие связей с чужими.
- **Интерфейсные элементы** – свои, имеющие связи с чужими.
- **Гало элементы** – чужие элементы, имеющие связи со своими (их копии хранятся у процесса для вычислений).

**Оптимизации:**
- **Блочное разбиение** (разбиение на блоки по обоим измерениям) выигрывает у строчного. Для квадратной решётки `N×N` и `P = p·p` процессов число граничных точек в блочном случае меньше, чем в строчном.
- **Переупорядочивание вершин** – улучшает локальность данных при работе с ними (вершины одного домена располагаются в памяти близко).
- **Асинхронные обмены MPI** – пока идёт расчёт на внутренних точках, можно выполнять обмен гало-элементами (использовать неблокирующие операции `MPI_Isend`/`MPI_Irecv`).
- **Объединение сообщений** – если пересылаются разные данные, лучше объединить их в одно сообщение, чтобы уменьшить накладные расходы.
- **Использование разделяемой памяти в узле** – если несколько процессов MPI работают на одном узле, можно создать специальный коммуникатор `MPI_COMM_TYPE_SHARED` для обмена через shared memory, сократив сетевой трафик.

### 4. Фундаментальные проблемы графовых алгоритмов на суперкомпьютерах

Эти проблемы ограничивают эффективность любых методов распараллеливания и характерны для нерегулярных графов (например, из курса Семенова):

- **Data-driven computations** – зависимость вычислений от данных (топологии графа). Невозможность применения методов статического распараллеливания вычислений.
- **Unstructured problems** – работа с нерегулярными, неструктурированными данными, трудность распараллеливания.
- **Poor locality** – низкая пространственно-временная локализация обращений к памяти (шаблон доступа непредсказуем, кэш малоэффективен).
- **High data access to computation ratio** – преобладание команд доступа к памяти над командами арифметических операций.

Из‑за этих особенностей даже на мощных суперкомпьютерах производительность графовых алгоритмов обычно составляет лишь единицы процентов от пиковой (в отличие от вычислительно интенсивных задач типа HPL).

## 5 билет: Архитектурные особенности графических процессоров, направленные на массивно-параллельные вычисления. Методы эффективной организации параллельных вычислений на графических процессорах.

### 1. Архитектурные особенности GPU

- **Огромное число вычислительных ядер** (несколько тысяч), объединённых в блоки. Ядра имеют общие элементы, в том числе регистры.
- **Простая архитектура ядра** – отсутствуют префетчеры, бранч‑предикторы и многие другие блоки, характерные для CPU.
- **Тактовые частоты GPU** ниже, чем у CPU, но благодаря большому количеству потоковых процессоров производительность (как пиковая, так и реальная) значительно выше для подходящих задач.
- **Память:**
    - **Высокая пропускная способность** видеопамяти (GDDR, HBM).
    - **Большая латентность** доступа к видеопамяти (на два порядка больше времени выполнения одной арифметической операции). Однако при хорошем параллелизме (много активных потоков) латентность перекрывается.
    - **Медленный обмен данными между оперативной памятью и видеопамятью** – канал PCI‑Express (или NVLink – проприетарное решение для коммуникации CPU‑GPU) является узким местом.
- **Аппаратная многопоточность** – если готовых к исполнению потоков мало, скрыть высокую латентность не удастся.
- **Разнородность архитектур** – оптимальные программы для NVIDIA (CUDA) и AMD (ROCm) будут сильно различаться.

### 2. Модель исполнения (SIMD / SIMT)

- **Минимальная единица исполнения** – **варп** (NVIDIA) или **ворк‑группа** (AMD). Нельзя выделить одно ядро; минимально будет задействован один варп (32 потока для NVIDIA).
- Все ядра внутри варпа выполняют **одни и те же инструкции**, но с разными данными – SIMD (NVIDIA использует термин **SIMT** – Single Instruction Multiple Threads).
- Из‑за простых логических блоков и общих регистров GPU **не любит ветвления** и сложной логики в алгоритмах.

### 3. Ограничения, накладываемые архитектурой

- Невозможность исполнить только одно ядро – операции группируются в варпы.
- Все потоки варпа выполняют одинаковую инструкцию; расходящиеся ветвления приводят к сериализации (выполняются сначала одна ветка, затем другая, с маскированием).
- Эффективность сильно зависит от шаблона доступа к памяти (коалесцинг, выравнивание).

### 4. Возможности и методы эффективной организации вычислений

- **Ускорение SIMD‑вычислений** – поэлементное сложение матриц, параллельная редукция, свёртки и т.д.
- **Выравнивание (alignment)** и **коалесцинг (coalescing)** – объединение обращений соседних потоков к памяти в одну транзакцию. Для достижения коалесцинга необходимо, чтобы потоки варпа обращались к последовательным адресам (например, при доступе к элементам массива с шагом 1).
- **Предвычисление всего, что можно** – вынос константных выражений, предварительное вычисление на CPU.
- **Замена деления умножением на обратное** (деление – дорогая операция).
- **Выбор между AoS и SoA** (Array of Structures vs Structure of Arrays) – для GPU обычно предпочтителен SoA, так как он даёт последовательный доступ и лучший коалесцинг.
- **Использование разделяемой памяти (shared memory)** – быстрое кэшируемое пространство внутри блока, позволяющее обмениваться данными между потоками блока.
- **Перекрытие латентности** – запуск большого количества варпов, чтобы во время ожидания данных одних варпов выполнялись другие.

## 6 билет: Методы статической и динамической балансировки загрузки процессоров: сдваивания, геометрического параллелизма, коллективного решения, конвейерного параллелизма, диффузной балансировки загрузки.

### 1. Причины несбалансированности вычислительной нагрузки

- Структура распределённого приложения неоднородна – различные логические процессы требуют разных вычислительных мощностей.
- Структура вычислительного комплекса (кластера) неоднородна – разные узлы обладают разной производительностью.
- Структура межузлового взаимодействия неоднородна – линии связи могут иметь различные пропускные способности.

### 2. Статическая балансировка

Выполняется **до** начала выполнения приложения. Часто использует опыт предыдущих выполнений (эвристики) или статическое размещение логических процессов по узлам. Недостаток: не может адаптироваться к динамическим изменениям (выход узлов из строя, фоновые процессы).

**Методы статической балансировки:**
- метод сдваивания;
- геометрический параллелизм;
- конвейерный параллелизм.

### 3. Динамическая балансировка

Перераспределение нагрузки **во время** выполнения приложения. Программное обеспечение определяет:
- загрузку вычислительных узлов;
- пропускную способность линий связи;
- частоту обменов сообщениями между логическими процессами.

**Методы динамической балансировки:**
- коллективное решение;
- диффузная балансировка загрузки.

### 4. Цель балансировки

Найти такое распределение задач по компьютерам (с учётом топологии сети), которое обеспечивает примерно равную вычислительную загрузку узлов и минимальные затраты на передачу данных между ними.

**Этапы решения задачи балансировки:**
1. Оценка загрузки вычислительных узлов.
2. Инициация балансировки загрузки.
3. Принятие решений о балансировке.
4. Перемещение объектов (задач или данных).

### 5. Метод сдваивания (каскадная схема)

Пример: найти сумму восьми элементов массива, используя 4 процессора.

- **Шаг 1:** на каждом процессоре найти сумму двух последовательных элементов.
- **Шаг 2:** на двух процессорах найти суммы результатов первого шага.
- **Шаг 3:** на одном процессоре получить итоговую сумму.

**Оценки времени, ускорения и эффективности** (для `n` элементов, `p = n/2` процессоров, `τ` – время одной операции):
- Время: `T_p(n) = τ * log2 n`.
- Ускорение: `S_p(n) = (n-1) / log2 n`.
- Эффективность: `E_p(n) ≈ 1 / log2 n` (падает с ростом n, так как процессоры задействованы только на первом шаге).

**Модифицированная каскадная схема** для повышения эффективности:
1. На первом этапе все суммируемые значения делятся на `n / log2 n` групп по `log2 n` элементов; для каждой группы последовательно вычисляется сумма.
2. На втором этапе для полученных `n / log2 n` сумм применяется обычная каскадная схема.

Временная оценка: `T_p = (n/p) * τ_c + (τ_c + τ_s) * log2 p`, где `τ_c` – время вычисления, `τ_s` – время передачи (синхронизации). При `p = log2 n` и `τ_s = 0`:
- Время ≈ `2 τ_c log2 n`.
- Ускорение ≈ `n / (2 log2 n)` = `p/2`.
- Эффективность ≈ `1/2`.

По сравнению с обычной каскадной схемой ускорение уменьшилось в 2 раза, но эффективность выросла почти до 0.5.

### 6. Геометрический параллелизм

Данные распределяются по процессорам равными частями, каждый процесс обрабатывает свою часть. Эффективен, когда действия процесса зависят лишь от ограниченного объёма данных с других процессоров.

**Пример задачи «Стена Фокса»:**
- Стена шириной `n`, высотой `k`. Кладка разбивается на равные участки, каждый участок строит отдельный каменщик (процесс).
- Каменщик перед укладкой очередного слоя должен убедиться, что на соседних участках предыдущий слой уже уложен. Иначе возникают паузы из‑за синхронизации: за `k` слоёв набегает `4k t_s` (два обмена с каждым соседом на слой).
- Время на одном процессоре: `T_1(kn) = τ_c * k * n`.
- Время на `p` процессорах: `T_p(kn) = τ_c * (k n / p) + 4 k τ_s`.
- Ускорение зависит от ширины стены `n`: для любого `τ_s` можно подобрать `n` так, чтобы эффективность была высокой.

**Условия применимости:**
- кирпичи распределены строго поровну;
- число операций на кирпич не зависит от его положения;
- все процессы работают с одинаковой скоростью;
- все каналы передачи данных имеют одинаковые характеристики.

Метод позволяет получать высокие ускорения при синхронной работе процессов с одинаковой скоростью.

### 7. Конвейерный параллелизм

Рассматривается задача с зависимостью по данным (каждый элемент зависит от элемента предыдущего слоя).

**Первый способ:** каждый процесс работает над своим слоем. Перед вычислением текущего элемента процесс получает элемент с предыдущего слоя, вычисляет свой, пересылает на следующий слой.  
*Недостаток:* низкая эффективность из‑за большого числа пересылок.

**Второй способ (геометрический параллелизм по вертикали):** работа (слои) делится между процессами. На каждом слое процесс только один раз принимает и передаёт данные. Эффективность высока при `p << k`. В противном случае накладные расходы на запуск и остановку конвейера снижают ускорение. Максимальная степень параллелизма = `min(n, k)`.

### 8. Метод коллективного решения

Сбор информации о загрузке всех узлов (коллективная операция), затем централизованное или распределённое принятие решения о перераспределении нагрузки. Формула перераспределения (из материала):

\[
n_i' = N \frac{\frac{n_i}{t_i}}{\sum_{j=0}^{p-1} \frac{n_j}{t_j}}
\]

где `n_i` – текущая нагрузка на процессе `i`, `t_i` – производительность процесса (или время выполнения единицы работы), `N` – общий объём работы, `n_i'` – новая нагрузка. Перераспределение стремится уравнять время выполнения.

### 9. Диффузная балансировка загрузки

Итеративный метод, при котором каждый процесс обменивается нагрузкой только с соседними процессами (по топологии сети). На каждой итерации избыток нагрузки перетекает к менее загруженным соседям. Процесс повторяется до достижения глобальной сбалансированности. Преимущество – локальность обменов, хорошо масштабируется. Недостаток – требуется много итераций при большой несбалансированности.

## 7 билет: Декомпозиция расчетных сеток. Критерии декомпозиции. Иерархические методы разбиения графов большого размера. Методы геометрической и спектральной бисекции, метод инкрементного роста, метод локального уточнения.

### 1. Критерии декомпозиции

1. **Классический критерий** – минимизация величины  
   \[ J = \max_{k=1,\dots,p} \sum_{v_i \in V_k} \left( w(v_i) + \alpha \sum_{v_j \in V_k} w(v_i, v_j) \right) \]  
   где `w(v_i)` – вес вершины, `w(v_i, v_j)` – вес ребра, `α` – коэффициент согласования единиц измерения. Критерий требует равномерного распределения суммарного веса узлов и минимизации максимального веса исходящих из домена рёбер.
2. Выделение обособленных доменов.
3. Минимизация максимальной степени доменов.
4. Обеспечение связности доменов.
5. Обеспечение связности множества внутренних узлов доменов.

### 2. Иерархические методы разбиения графов большого размера

Этапы:
1. **Огрубление (coarsening)** – построение последовательности уменьшающихся вложенных графов.
2. **Начальная декомпозиция** – распределение вершин огрублённого графа по заданному числу доменов.
3. **Восстановление и локальное уточнение** – перераспределение вершин всех графов последовательности по доменам.

**Огрубление** – итерационное стягивание рёбер. На каждой итерации выбирается множество рёбер покрытия так, чтобы никакая вершина не была инцидентна более чем одному выбранному ребру (предпочтение отдаётся рёбрам между вершинами с наименьшими весами). Выбранное ребро стягивается, получается новая вершина; веса новой вершины и новых рёбер равны сумме весов стянутых. Процесс повторяется до получения графа малого размера.

**Начальная декомпозиция** огрублённого графа может выполняться:
- сразу на требуемое число доменов;
- рекурсивно (рекурсивная бисекция);
- спектральным методом.

**Восстановление** – обратный процесс: граф восстанавливается до исходного размера, при этом используется **локальное уточнение** для перераспределения вершин между доменами.

### 3. Метод локального уточнения (алгоритм Кернигана‑Лина, KL)

Эвристический алгоритм, который проверяет выигрыш от перемещения вершины в соседние домены. Выигрыш – уменьшение общего веса разрезанных рёбер, полученного от перемещения вершины.

**Итеративный процесс:**
1. Для каждой вершины определяется выигрыш от перемещения в соседние домены.
2. Определяются пары «вершина‑домен» с большими выигрышами.
3. Временно переносятся вершины с положительным выигрышем в новые домены.
4. Если величина `J` уменьшилась, новое разбиение принимается как текущее и выполняется следующая итерация.
5. Если `J` не уменьшилась, изменения отменяются и выбирается другое множество перемещаемых вершин.
6. Если несколько попыток выбрать множество вершин закончились неуспехом, разбиение считается оптимальным и процесс заканчивается.

### 4. Геометрическая бисекция

Метод основан на рекурсивной координатной бисекции. На каждом этапе рекурсии область (сетка) разбивается на две части прямой (по рёбрам сетки). Полученные подобласти разбиваются дальше аналогично, пока в каждой не останется по одному домену.

**Шаг координатной бисекции:**
- Вершины сортируются по возрастанию одной из координат.
- В первую группу относят вершины с меньшим значением координаты.
- Предварительное упорядочивание происходит по всем вершинам.
- Разрезание выполняется по той координате, которая даёт наименьшее количество разрезанных рёбер (можно учитывать расположение узлов и по другим координатам).

### 5. Спектральная бисекция

Разбивает граф на 2 домена, минимизируя вес рёбер между доменами. Вычислительно затратный метод, чаще применяется для огрублённой сетки.

**Математическая постановка:**  
Пусть `q[i] = -1` или `1` определяет принадлежность вершины `i` одному из двух доменов. Число разрезанных рёбер:

\[ |E_c| = \frac{1}{4} \sum_{i,j \in E} (q_i - q_j)^2 \]

Необходимо минимизировать `|E_c|` при условиях:
1) равенство числа вершин в доменах: `∑ q_i = 0` (нормировка);
2) `∑ q_i^2 = q^T q = |V|`.

**Этапы решения:**
- **А. Построение спектральной матрицы графа (матрицы Лапласа)**:  
  Элементы `l_{ij}`:  
  `l_{ij} = -1` если `e_{ij} ∈ E, i≠j`,  
  `l_{ii} = ∑_{k≠i} l_{ik}` (степень вершины),  
  `l_{ij} = 0` иначе.
- **Б. Определение собственного вектора Фидлера**:  
  Решаем `L q = λ q`. Матрица Лапласа вырождена (`det(L)=0`), одно собственное значение `λ_0 = 0` с вектором из единиц. Минимальное ненулевое собственное значение `λ_1` соответствует собственному вектору `ψ` (вектор Фидлера). `ψ` ортогонален единичному вектору: `e ψ = 0`.
- **В. Упорядочивание вершин**: сортируем номера вершин по возрастанию значений элементов `ψ`, получаем вектор `s`.
- **Г. Формирование двух доменов**: распределяем вершины в пропорции `Q = n1/(n1+n2)`. Первая часть содержит вершины из первой половины `s`, вторая – из второй половины.

### 6. Метод инкрементного роста

Основан на понятиях **граничная вершина** (имеет соседа из другого домена или является граничной всей сетки), **глубина вершины** (кратчайшее расстояние до любой граничной вершины), **ядро графа R** (подграф из вершин с глубиной не менее `k` и инцидентными рёбрами), **слой T_k** (подграф из вершин глубины `k` и инцидентных рёбер).

**Алгоритм:**
1. **Инициализация доменов** – определение `p` случайно выбранных вершин.
2. **Распределение вершин по доменам** методом инкрементного роста: к каждому домену последовательно добавляются слои вершин (ближайшие к границе).
3. **Локальное уточнение** границ сформированных доменов (KL‑алгоритм).
4. **Анализ связности ядер** сформированных доменов. Если заданный уровень связности достигнут – конец; иначе – переход к шагу 5.
5. **Перенос** части закреплённых за доменами вершин в группу свободных вершин (удаление нескольких внешних слоёв или всех слоёв доменов, не удовлетворяющих связности), переход к шагу 2.

Метод сохраняет высокое качество разбиения, усиливает компактность доменов, но сокращение общего числа разрезанных рёбер не гарантируется.

## 8 билет: Параллельные алгоритмы сортировки данных.

### 1. Постановка задачи

Расположить в порядке неубывания `N` элементов массива чисел, используя `p` процессоров.

**Два варианта:**
1. Объём оперативной памяти одного процессора достаточен для размещения всех `N` элементов.  
   *Суть:* расположить элементы массива `a` так, чтобы `a_i ≤ a_{i+1}`.
2. Объём памяти одного процессора мал для одновременного размещения всех элементов.  
   *Условия:* каждая часть массива хранится на отдельном процессоре; каждая часть должна быть упорядочена; на процессорах с большими номерами должны быть элементы с большими значениями.  
   *Стратегия:* упорядочивание фрагментов на каждом процессоре, затем перераспределение элементов между процессорами.

### 2. Последовательные алгоритмы

Наилучший последовательный алгоритм – **dsort** (комбинация hsort и dsort):
- **hsort** (пирамидальная сортировка) – для коротких фрагментов (плох на больших объёмах из‑за нелокальности доступа и кэш‑промахов).
- **dsort** – алгоритм сортировки слиянием.

**Слияние двумя способами:**
- **Стандартное (одним процессом):** один процесс проходит от меньших элементов к большим и сливает два упорядоченных массива по `k` элементов за `2k` тактов.
- **Двустороннее слияние (двумя процессами):** один процесс идёт от меньших к большим, другой – от больших к меньшим; тактов в 2 раза меньше.

### 3. Параллельные алгоритмы сортировки

#### 3.1. Слияние методом сдваивания

Нерекурсивный алгоритм `dsort` допускает параллельную обработку на основе метода сдваивания. Этот метод и двустороннее слияние эффективно используют большое количество процессоров, но не обеспечивают обработку данных, размер которых превышает объём памяти одного узла.

#### 3.2. Сети сортировки

Сеть сортировки – алгоритм, в котором порядок и количество операций сравнения не зависят от значений элементов. Каждый элемент обрабатывается **компараторами** (сравнение/перестановка).

**Представление:**  
- Сортируемые элементы – горизонтальные линии данных.  
- Компараторы – вертикальные отрезки, соединяющие две линии.  
- Компаратор имеет 2 входа и 2 выхода; после сравнения на верхнем выходе оказывается меньший элемент, на нижнем – больший.

**Сеть обменной сортировки со слиянием Бэтчера** – наиболее быстрая из масштабируемых сетей сортировки.

**Алгоритм для массива из `n` элементов:**
1. Разделить массив на две части.
2. Отсортировать каждую часть.
3. Объединить результаты с помощью `(n, m)` сети **нечётно-чётного слияния Бэтчера**.

**Чётно-нечётное слияние Бэтчера:**
- Отдельно сливаются элементы с нечётными номерами из обоих массивов, отдельно – с чётными.
- Затем с помощью заключительной группы компараторов обрабатываются пары соседних элементов с номерами `(2i, 2i+1)` для `i` от `1` до `n/2-1`.

**Общая структура алгоритма** (рекурсивная):
- `odd-even-merge(n, m)` – слияние двух упорядоченных последовательностей длины `n` и `m`.
- Сначала рекурсивно выполняется `odd-even-merge(n/2, m/2)` для нечётных и чётных индексов, затем применяются компараторы между соседними элементами.

**Оценка числа операций:**  
Для слияния двух массивов длины `n` требуется `O(n log n)` сравнений. Полная сортировка сети Бэтчера для `n` элементов имеет глубину `O(log^2 n)` и использует `O(n log^2 n)` компараторов. Это асимптотически оптимально для неадаптивных сетей сортировки.

### 4. Применимость параллельных алгоритмов сортировки

- **Слияние методом сдваивания** и **двустороннее слияние** хороши для массивов, помещающихся в память одного узла, и для большого числа процессоров.
- **Сети сортировки** (особенно Бэтчера) обеспечивают детерминированное время выполнения и могут быть эффективно реализованы на параллельных машинах с фиксированной топологией (например, гиперкуб), но требуют априорного знания количества элементов.

Для распределённой памяти (вариант 2) применяется стратегия: локальная сортировка фрагментов + глобальное перераспределение (например, с помощью параллельной сортировки слиянием или метода «транспозиции» на торе). Конкретные реализации зависят от топологии сети и объёма данных.

## 9 билет: Лямбда-архитектура обработки больших данных

### 1. Пакетная и потоковая обработка данных

**Пакетная обработка** – работа с большим фиксированным объёмом данных.  
Пример: хранилище данных, куда постоянно добавляются новые данные; обработка запускается, например, раз в сутки (над всем объёмом накопленных данных либо над большим куском – неделя, месяц,…).  
Данные, которые пришли во время очередного прогона пакетной обработки, не будут учтены – нужно ждать сутки.  
Основной недостаток пакетной обработки – **задержка (latency)** и **несвежие данные**.  
Из‑за этого возникают проблемы согласованности (consistency). Например, при вычислении статистики по фирмам, если во время прогона пакетной обработки одна из фирм меняет название, статистические показатели могут быть приписаны несуществующей фирме.

**Потоковая обработка** – поток данных бесконечен, в каждый момент времени имеем дело с некоторым окном.  
**Поток событий** – неограниченный набор данных (записей о произошедших событиях – изменение цены акции, публикация сообщения в соцсети и т.п.), снабжённых временными метками.  
Поток упорядочен, порядок событий имеет значение.  
Пример: на счёте 0 руб. последовательность {положить 100 руб, потратить 100 руб} отличается от {потратить 100 руб, положить 100 руб} (во втором случае – комиссия за перерасход).

#### 1.1. Время в потоковой обработке
Событию может приписываться временная метка:
1. при возникновении события (датчик температуры);
2. при добавлении данных в хранилище (но данные могут приходить неравномерно, искажая средние);
3. время обработки (худший случай: обработка на разных узлах, часы могут расходиться; если поток идёт через несколько обрабатывающих узлов, одно и то же событие может получить разные метки).

#### 1.2. Состояние (state)
Любые промежуточные данные, необходимые для обработки потока (вычисление максимума, среднего и т.д.).  
Виды состояний:
- **локальное** – доступно только одному экземпляру приложения, потребляет ресурсы приложения;
- **внешнее** – хранится на специальных узлах хранения в СУБД (Cassandra); пространство для хранения практически не ограничено, но обращение к внешнему состоянию добавляет задержку; нужны алгоритмы, обращающиеся к состоянию как можно реже.

#### 1.3. Свойства потока
- **Неизменяемость записей** – если товар продан, а затем возвращён, событие «продажа» не удаляется; вместо этого в поток добавляется событие «возврат».  
  В СУБД журналы операций (WAL, redo‑log) – аналоги потоков.
- **Дуализм потока и таблицы** – поток событий может быть превращён в таблицу (применение потока операций к начальным данным), а таблица – в поток (история изменений данных).  
- **Повторяемость (recomputable, replayable)** – поток может быть воспроизведён заново.

#### 1.4. Особенности потоковой обработки
- обрабатываются небольшие объёмы данных (окна длиной несколько минут);
- применяются инкрементальные, часто приближённые вероятностные алгоритмы;
- особые требования к хранилищам результатов – **представления реального времени (real‑time views)**.

**Виды окон:**
- **последовательные** (12:10:00–12:14:59, 12:15:00–12:19:59,…);
- **накладывающиеся** (12:10:00–12:14:59, 12:11:00–12:15:59);
- **скользящие** – сдвиг окна фиксированной ширины на одну точку.

Основной недостаток потоковой обработки – **отсутствие возможности посмотреть старые данные**.

### 2. Лямбда-архитектура (Lambda Architecture)

Гибрид потоковой и пакетной обработки, устраняющий недостатки каждого подхода.

**Основные компоненты:**
- **пакетный слой** – работает со всем массивом накопленных данных, регулярное перевычисление (recompute).  
  Пример: пришли данные за последние сутки по температуре, перевычисляем общее среднее (за всю историю наблюдений).  
  Данные неизменяемы → перевычисления делать легко, старые значения не перезаписываются.  
- **потоковый слой** – обрабатывает свежие данные в реальном времени, давая приближённые результаты с малой задержкой.

Пакетная обработка выполняется:
- по расписанию (учитываются добавленные данные за последние сутки);
- при изменении кода, выполняющего обработку.

**Разница в точности вычислений:**
- пакетный уровень: точные алгоритмы (например, подсчёт количества уникальных элементов традиционным алгоритмом);
- потоковый уровень: приближённые алгоритмы (например, HyperLogLog для подсчёта уникальных элементов).

### 3. Реализации лямбда-архитектуры

Лямбда-архитектура не привязана к конкретному ПО (product‑agnostic); может быть собрана из разных продуктов, в том числе свободно распространяемых.

**Виды компонент:**
1. **Пакетные системы обработки данных** – HDFS + MapReduce.  
   HDFS: масштабируемость, отказоустойчивость, хранение петабайтов данных.  
   MapReduce: горизонтальная масштабируемость, работа с HDFS.
2. **ПО для сериализации** – Avro, Apache Thrift, Google ProtoBuf. Передача объектов между компонентами на разных языках программирования.
3. **NoSQL СУБД** – простая модель данных и языки запросов, подбираются под конкретный вид данных.
4. **Системы передачи сообщений** – очереди RabbitMQ, Apache Kafka; также RPC, REST.
5. **Потоковые системы обработки данных** – Apache Storm.

### 4. Достоинства и недостатки лямбда-архитектуры

**Достоинства:**
- использование неизменяемых (immutable) данных;
- возможность выполнения повторной обработки данных с нуля (recompute).

**Мнимые достоинства (опровергнуты в материале):**
- «Λ‑архитектура даёт возможность дополнить приближённые результаты потоковой обработки точными результатами пакетной обработки» – не является строгим преимуществом;
- «За счёт объединения двух систем – потоковой и пакетной, с различными компромиссами по согласованности и доступности, Λ‑архитектура даёт возможность преодолеть ограничения CAP‑теоремы» – на самом деле это не так.

**Недостатки:**
1. сложность;
2. необходимость поддерживать две кодовые базы – для потоковой и для пакетной обработки.

## 10 билет: Консенсус в распределённых системах

### 1. Задачи координации в распределённых системах
- выбор лидера;
- взаимное исключение;
- снимок глобального состояния.

Эти задачи сводятся к **задаче достижения консенсуса**.

### 2. Распределённые алгоритмы

Каждый узел принимает решение на основе **локальной информации** о текущем состоянии системы (информация может быть частично устаревшей).  
Не используются глобальная информация, глобальное время, координатор.

**Основные свойства распределённого алгоритма:**
- **живость (liveness)** – алгоритм движется вперёд, задача будет решена в конце концов; «что‑то хорошее обязательно произойдёт».
- **безопасность (safety)** – нежелательная ситуация не возникнет ни при каких условиях; «что‑то плохое никогда не произойдёт».
- иногда добавляют **справедливость**.

### 3. Общие алгоритмы консенсуса

Наиболее общая формулировка: все процессы должны договориться о значении некоторой переменной.

**Требования к алгоритмам консенсуса:**
- **завершаемость (живость)** – каждый корректный процесс принимает некоторое решение;
- **согласие (безопасность)** – все корректные процессы приходят к одному решению;
- **истинность (безопасность)** – принятое значение должно было быть предложено одним из узлов.

Наиболее известные протоколы консенсуса: **Paxos, Raft, ZAB**.  
Эти алгоритмы **не устойчивы к византийским отказам**.  
Устойчивы к византийским отказам: **PBFT (practical Byzantine fault tolerance), proof‑of‑work**.

### 4. Теорема FLP (Фишер‑Линч‑Патерсон)

Утверждение о невозможности в асинхронных системах:

> В асинхронной системе при возможности отказа хотя бы одного узла невозможно создать детерминированный алгоритм консенсуса, который обладал бы одновременно свойствами **живости**, **безопасности** и **отказоустойчивости**.

### 5. Обработка отказов координатора

Многие распределённые системы используют выделенный узел‑координатор. При его падении система становится неработоспособной.  
Необходим механизм, который автоматически сделает один из оставшихся узлов новым координатором – **алгоритмы выбора лидера**.

### 6. Обнаружение отказов

**Детектор отказа (failure detector)** – абстрактный функционал, обеспечивающий обнаружение отказа процесса (алгоритм на узлах либо внешний сервис).

**Основные механизмы:**
- **контроль со стороны системы** – процессы шлют наблюдаемому процессу эхо‑запросы (ping);
- **пульсация (heartbeat)** – наблюдаемый процесс сам шлёт контрольные пакеты.

Детектор отказа может ошибаться. Для уменьшения вероятности ошибки процесс, обнаруживший отказ, может «попросить» другие процессы послать контрольный пакет подозрительному узлу – тогда факт отказа становится предметом консенсуса группы процессов.  
- В синхронной системе обычно используется **таймаут**;
- в асинхронной системе – счётчики контрольных пакетов.

### 7. Выбор лидера (алгоритм Гарсии‑Молины)

Работает в предположениях синхронности системы, надёжности сети, возможности отказа процессов.  
Каждый процесс знает общее количество процессов и их идентификаторы, но не знает их текущего состояния. Требуется полная топология сети.

**Алгоритм:**
1. Процесс P запускает выборы, если подозревает падение лидера.
2. P шлёт сообщение **ELECTION** процессам с **большими** id.
3. Если получает в ответ **OK** – становится пассивным, ждёт окончания выборов.
4. Если не получает ответов – становится лидером, рассылает всем сообщение **COORDINATOR**.
5. Процесс, получив ELECTION от процесса с меньшим id, шлёт в ответ OK и запускает свой выборный процесс.  
   В случае перезапуска бывшего (сломавшегося) лидера он начинает выборы.  
   Если несколько процессов заметят падение лидера и начнут выборы одновременно, в системе будет параллельно действовать несколько волн выборов. Однако в результате лидером всё равно станет процесс с максимальным id.

**Количество служебных сообщений (message complexity)** – **O(N²)**.  
В случае одного инициатора – реализуется, когда его id близок к N; в случае, когда все являются инициаторами – всегда O(N²).

### 8. Глобально упорядоченная (атомарная) рассылка

Система, в которой каждый процесс рассылает остальным широковещательные сообщения (например, реплицированная БД).  
Необходимо, чтобы все сообщения на всех узлах были упорядочены одинаково с соблюдением причинно‑следственных связей.

**Предположения:**
- каналы обладают свойством **FIFO**;
- каждый процесс имеет локальную очередь (буфер) сообщений.

**Механизм:**  
Каждый процесс, получив сообщение, помещает его в очередь и поддерживает упорядоченность по меткам времени (L(m), i), где i – номер процесса.  
Процесс доставляет сообщение m, стоящее первым в его локальной очереди, при условии, что от всех остальных процессов получены сообщения с меткой **больше**, чем у m (это гарантирует, что ни в одном канале нет сообщений с меньшими метками, благодаря FIFO).  
Для гарантии прихода сообщений с большей меткой используются **подтверждения**: процесс, получив сообщение, рассылает подтверждение всем остальным процессам; подтверждение имеет логическую метку больше, чем сообщение, на которое оно было послано.

**Применение:** атомарная рассылка (реализована в протоколе **ZAB – Zookeeper Atomic Broadcast**) используется в распределённом сервисе координации **Zookeeper**, обеспечивающем отказоустойчивость кластеров Hadoop.

### 9. Взаимное исключение

**Централизованный алгоритм (с координатором):**  
- Рабочий узел для входа в критическую секцию шлёт **REQUEST** координатору, ожидает разрешения; при выходе шлёт **RELEASE**.  
- Координатор поддерживает очередь запросов, на которые ещё не ответил OK, шлёт OK процессу в голове очереди, удаляет его при получении RELEASE.

**Алгоритм Лэмпорта (без координатора):**  
У каждого процесса своя очередь запросов Q.
1. Процесс, желающий войти в КС, рассылает всем (включая себя) сообщение **request** с отметкой времени.
2. При получении request процесс помещает его в свою Q.  
   a. Если процесс не находится в КС, он посылает отправителю подтверждение **ack**.  
   b. Иначе откладывает отправление подтверждения до выхода из КС.
3. Процесс входит в КС, если:  
   a. его запрос имеет **наименьшую отметку времени** среди всех запросов в Q;  
   b. он получил подтверждения на свой запрос от всех остальных процессов.
4. Перед выходом из КС процесс удаляет свой запрос из своей Q и посылает всем **release**.
5. При получении release процесс удаляет соответствующий запрос из своей Q.

### 10. Распределённые транзакции. Алгоритм двухфазной фиксации (2PC)

Транзакция фиксируется только если все реплики смогли её выполнить; иначе – откат на всех репликах.

**Алгоритм 2PC** (требует процесса‑координатора):

**Фаза 1:** координатор связывается с участниками (репликами) с вопросом «Можешь ли ты быстро и гарантированно завершить транзакцию?»  
- Если реплика может (её ресурсы не заблокированы) – блокирует ресурсы, отвечает **OK**.  
- Иначе – отвечает **NO**.

**Фаза 2:** после сбора откликов:  
- если все ответили OK – координатор принимает решение **зафиксировать** транзакцию;  
- если хотя бы одна ответила NO – **откатить**.  
Координатор связывается с репликами и приказывает выполнить решение.

### 11. Алгоритм ZAB (Zookeeper Atomic Broadcast)

Протокол консенсуса, используемый в **ZooKeeper** – высокодоступном, высокопроизводительном, распределённом сервисе координации (фактически СУБД «ключ‑значение» с особым API). Архитектура **master‑slave**.

**Требования к протоколу широковещательной рассылки:**
- **надёжная доставка** – если сообщение m доставлено одним сервером, то в конце концов оно будет доставлено всеми корректными серверами;
- **глобальное упорядочивание** – если на одном сервере сообщение a доставлено раньше b, то каждый сервер, доставляющий оба, доставляет a раньше b;
- **сохранение причинно‑следственных связей** – если a предшествует b, то a доставляется раньше b;
- свойство FIFO обеспечивается TCP;
- если m – последнее сообщение, доставленное от лидера L, каждое сообщение, предложенное до m лидером L, должно быть доставлено.

**Основные этапы протокола Zab:**
1. выбор лидера;
2. обнаружение реплик;
3. синхронизация;
4. рассылка обновлений.

Фазы 1‑3 выполняются после сбоя лидера или при запуске кластера (фазы 2 и 3 часто объединяют в **фазу восстановления**).

**Нормальная работа кластера ZooKeeper:**
- клиент подключается к случайному узлу кластера;
- запрос на чтение выполняется с этого узла;
- запрос на запись: если подключились к мастеру – обрабатывается мастером; если к ведомому – перенаправляется мастеру.
- мастер транслирует транзакцию на узлы кластера; операция завершается, если больше половины узлов зафиксируют транзакцию.

**Понятие эпохи:** при смене лидера наступает новая эпоха (счётчик увеличивается на единицу).  
**Рассылка сообщений:** мастер создаёт предложение, присваивает глобальный номер транзакции **ZXID**, рассылает предложение другим узлам. На мастере есть очереди рассылаемых сообщений (по одной на каждого последователя); предложения помещаются в очереди, затем рассылаются в соответствии с FIFO.

**Восстановление после отказа:**  
Если последователь не может подключиться к лидеру, он переключается в состояние **LOOKING** и инициирует новый раунд выборов.  
Если мастер не может должным образом взаимодействовать более чем с половиной серверов, он также переключается в LOOKING, уступая лидерство узлу с лучшей сетевой средой.  
Восстановление состоит из двух этапов: выбор лидера и восстановление данных.

### 12. Консенсус в биткоине: Proof‑of‑Work (PoW)

Используется для достижения консенсуса в децентрализованной сети без доверенных узлов. Участники (майнеры) решают вычислительно сложную задачу (нахождение хеша, меньшего заданной цели), что требует затрат ресурсов. Первый, кто нашёл решение, предлагает новый блок; остальные проверяют решение и принимают блок, удлиняя цепочку. Атака требует контроля более 50% вычислительной мощности сети.

## 11 билет: Корпоративные хранилища данных (Data Warehouse), озёра данных

### 1. Хранилище данных (Data Warehouse, DWH)

Предметно‑ориентированная, интегрированная, неизменчивая и поддерживающая хронологию система сбора и хранения данных, предназначенная для поддержки принятия управленческих решений.  
В отличие от операционных баз данных (OLTP), DWH оптимизировано для сложных аналитических запросов над большими объёмами исторических данных (OLAP).

**Ключевые свойства DWH (по Имону):**
- **Предметная ориентированность** – данные организованы вокруг ключевых бизнес‑сущностей (клиенты, продукты, продажи), а не вокруг прикладных процессов.
- **Интегрированность** – данные собираются из множества разнородных источников (OLTP‑системы, внешние файлы) и приводятся к единому формату.
- **Неизменчивость (стабильность)** – данные не обновляются и не удаляются, только добавляются (append‑only), сохраняется история изменений.
- **Поддержка хронологии** – каждый факт снабжается временной меткой, позволяющей анализировать данные в разрезе времени.

### 2. Два классических подхода к построению DWH

#### 2.1. Концепция Иимона (top‑down, централизованное хранилище)
- Создаётся единое **корпоративное хранилище данных (EDW – Enterprise Data Warehouse)**, «единственный источник правды» (Single Source of Truth).
- Данные в EDW хранятся в строго нормализованной форме (обычно 3НФ) – минимизация избыточности, гарантия целостности.
- Используется подход **Schema‑on‑Write**: данные очищаются, трансформируются и проверяются до загрузки в хранилище (ETL).
- На основе EDW создаются **зависимые витрины данных** для нужд конкретных подразделений (часто в денормализованном виде – схема «звезда»).
- *Преимущества*: высокое качество данных, гибкость для будущих аналитических задач.  
  *Недостатки*: высокая начальная сложность, длительное время внедрения.

#### 2.2. Концепция Кимбалла (bottom‑up, объединение витрин)
- Хранилище данных – совокупность взаимосвязанных **витрин данных (data marts)**, каждая обслуживает конкретный бизнес‑процесс.
- Витрины связаны через **архитектуру шины (Data Warehouse Bus)** с использованием **согласованных измерений (conformed dimensions)** – одинаковой структуры и значений во всех витринах.
- Данные организуются в многомерные модели: **схема «звезда»** (центральная таблица фактов с мерами и денормализованные таблицы измерений) или **схема «снежинка»** (нормализованные иерархии измерений).
- Ориентация на бизнес‑процессы позволяет быстро получать результаты итеративно.
- *Преимущества*: быстрое внедрение, простота для конечных пользователей, высокая производительность запросов.  
  *Недостатки*: риск возникновения изолированных витрин при несоблюдении шины.

### 3. Витрины данных (Data Marts)

Специализированное подмножество хранилища, ориентированное на конкретный отдел или предметную область (продажи, маркетинг, финансы).  
**По источнику данных:**
- **Зависимая витрина** – данные из центрального EDW (подход Инмона), единый источник истины.
- **Независимая витрина** – данные напрямую из операционных систем, быстрое решение локальных задач, но риск создания «лоскутных» хранилищ (data silos).
- **Гибридная витрина** – сочетает данные из EDW и внешних источников.

### 4. Озёра данных (Data Lake)

Централизованное хранилище для огромных объёмов данных в их исходном, «сыром» (raw) формате. Может содержать структурированные (таблицы), полуструктурированные (JSON, XML, CSV) и неструктурированные (изображения, видео, логи) данные.

**Ключевые характеристики Data Lake:**
- **Schema‑on‑Read** – схема данных определяется только в момент чтения и анализа; гибкость интерпретации.
- Хранение любых типов данных без предварительной обработки – идеально для Data Science и машинного обучения.
- **ELT (Extract, Load, Transform)** – данные сначала загружаются «как есть», преобразование выполняется позже по требованию.
- Горизонтальное масштабирование на основе распределённых файловых систем (HDFS) или облачных хранилищ (Amazon S3, ADLS, GCS).
- Риск **«болота данных»** – без управления (каталогизация, контроль качества, data governance) озеро превращается в хаотичную свалку.

### 5. Lakehouse (Озеро‑дом)

Гибридная архитектура, объединяющая достоинства озёр данных (гибкость, экономичность) и хранилищ данных (ACID‑транзакции, производительность).  
Ключевой элемент – добавление над файлами озера (обычно в формате Parquet) слоя транзакционных метаданных (например, **Delta Lake, Apache Iceberg**). Этот слой обеспечивает:
- ACID‑транзакции при одновременном чтении и записи;
- управление версиями (time travel) – доступ к состоянию данных на любой момент времени;
- единую платформу для BI‑отчётов, SQL‑запросов и задач машинного обучения.

**Сравнение архитектур (таблица из материала):**

| Характеристика                | Data Warehouse (DWH) | Data Lake (Озеро данных) | Lakehouse                     |
|-------------------------------|----------------------|--------------------------|-------------------------------|
| Типы данных                   | преимущественно структурированные | любые (структурированные, полу‑ и неструктурированные) | любые |
| Схема                         | Schema‑on‑write      | Schema‑on‑read           | Schema‑on‑write с эволюцией   |
| Транзакции                    | полная поддержка ACID | нет ACID (уровень файлов) | поддержка ACID                |
| Производительность            | высокая для BI и отчётности | высокая для ML, низкая для BI | высокая для любых типов нагрузок |
| Хранение                      | дорогое специализированное | дешёвое облачное объектное | дешёвое облачное объектное     |
| Основные сценарии             | бизнес‑аналитика, отчётность | Data Science, ML, архивирование | все сценарии: BI, отчёты, ML |

**Apache Parquet** – де‑факто стандарт хранения в озёрах и Lakehouse. Колоночный бинарный формат с эффективным сжатием (словарное, RLE, битовая упаковка) и предикатным пушдауном (пропуск блоков данных на основе метаданных min/max и фильтров Блума), что ускоряет аналитические запросы.

## 12 билет: Клеточные автоматы: определение, элементарные клеточные автоматы, классификация Вольфрама, двумерные клеточные автоматы, типы окрестностей, игра «Жизнь», параллельная реализация

### 1. Определение клеточного автомата

Клеточные автоматы введены Джоном фон Нейманом в 1940‑х годах для моделирования самовоспроизведения в биологических системах. Обладают высокой степенью внутреннего параллелизма.

**Неформальное определение:**
- Регулярно упорядоченный набор (конечный или бесконечный) простых однотипных объектов – **клеток**.
- Каждая клетка обладает **внутренним состоянием**; множество возможных состояний дискретно и конечно.
- Клетки обновляют состояния **одновременно (синхронно)** в дискретные моменты времени.
- **Правила обновления** однородны (не зависят от времени и пространственного расположения клетки).
- Новое состояние клетки **детерминированно** определяется состояниями клеток, входящих в её **локальную окрестность**.

### 2. Элементарные клеточные автоматы

Одномерные, двоичные (`m = 2`), ранг `r = 1`.  
Клетки располагаются в целочисленных узлах координатной прямой. Локальная окрестность i‑й клетки содержит клетки `i-1`, `i`, `i+1` (всего 3 клетки).

**Число клеток** может быть бесконечным или конечным. Для конечного отрезка `[0, n-1]` вводятся **граничные условия**:
- фиксированные состояния граничных клеток;
- циклические условия (отрезок сворачивается в цикл).

Всего элементарных автоматов: \(2^{2^3} = 2^8 = 256\).  
Каждому автомату ставится в соответствие **номер** – двоичное число, полученное из таблицы истинности (последний столбец, записанный снизу вверх). Пример: автомат №90 (01011010₂).

### 3. Классификация Вольфрама (1980‑е)

Стивен Вольфрам проанализировал все 256 автоматов и выделил 4 типа поведения:

1. **1‑й тип** – за конечное число шагов достигают пространственно‑однородного состояния и перестают эволюционировать.  
   *Примеры:* правила 0, 32, 160, 232.
2. **2‑й тип** – генерируют локализованные структуры, стационарные или периодические по времени.  
   *Примеры:* правила 4, 108, 218, 250.
3. **3‑й тип** – хаотическое поведение даже при упорядоченном начальном состоянии.  
   *Примеры:* правила 22, 30, 126, 150, 182.
4. **4‑й тип** – смесь хаотического и детерминированного поведения; возникают взаимодействующие структуры.  
   *Пример:* правило 110 (полное по Тьюрингу, алгоритмически универсально).  
   *Неоднозначные случаи:* правила 62, 73 (зависимость от начального состояния), правило 54 (неизвестна полнота по Тьюрингу, но относят к 4 классу).

### 4. Двумерные клеточные автоматы

Клетки располагаются на плоскости в точках с целочисленными координатами (прямоугольная решётка).

**Типы локальных окрестностей (ранг r = 1):**
- **Окрестность фон Неймана** – соседями являются клетки, имеющие общее ребро (север, юг, запад, восток) – 4 соседа.
- **Окрестность Мура** – соседями являются клетки, имеющие общее ребро или общую вершину (8 соседей – все вокруг).

Для двоичных двумерных автоматов число наборов правил огромно:
- окрестность фон Неймана: \(2^{2^5} = 2^{32}\);
- окрестность Мура: \(2^{2^9} = 2^{512}\).

### 5. Игра «Жизнь» (Conway's Game of Life)

Двоичный двумерный клеточный автомат с окрестностью Мура ранга 1.  
**Интерпретация состояний:**
- 1 – «живая» клетка (закрашенный квадрат);
- 0 – «мёртвая» клетка (незакрашенный квадрат).

**Правила эволюции:**
- живая клетка, у которой **меньше двух живых соседей**, умирает («одиночество»);
- живая клетка, у которой **больше трёх живых соседей**, умирает («перенаселённость»);
- мёртвая клетка, у которой **ровно три живых соседа**, становится живой;
- во всех остальных случаях состояние не меняется.

Автомат Конвея относится к **4‑му типу** по Вольфраму. Любая случайная начальная конфигурация спустя (обычно) небольшое число шагов приводит к появлению простых устойчивых объектов – стационарных, периодических или движущихся.  
Простейший движущийся объект – **глайдер (планер)** – состоит из пяти живых клеток.

### 6. Параллельная реализация (MPI)

В материале приведён алгоритм для параллельного моделирования на MPI. Основные шаги:

1. Процесс `rank = 0`:
   - вычисляет размер `p = |√size - 1|`, `N = p·n`;
   - создаёт двумерный массив `data[0..N+1, 0..N+1]`;
   - задаёт начальное состояние автомата;
   - вызывает `SETUPBOUNDARY(data, N)`.

2. Остальные процессы создают массив `data[0..n+1, 0..n+1]` и распределяют данные (`DISTRIBUTE DATA`).

3. Цикл по времени `t` от 1 до `T`:
   - для процессов `rank ≠ 0` – обновление состояния (`UPDATE DATA`);
   - вызов `SETUPBOUNDARY MPI(data, n, p)` – обмен граничными строками и столбцами с соседними процессами;
   - сбор данных на процесс 0 (`COLLECT DATA`).

**Функция `SETUPBOUNDARY MPI(data, n, p)`** (в псевдокоде):  
Определяются номера соседних процессов:
- `left` = `rank-1` если `j ≠ 0`, иначе `rank+p-1`;
- `right` = `rank+1` если `j ≠ p-1`, иначе `rank-p+1`;
- аналогично `above` и `below`.

Затем выполняются обмены:
- `MPI_SENDRECV(data[*, 1], left, data[*, n+1], right)`
- `MPI_SENDRECV(data[*, n], right, data[*, 0], left)`
- `MPI_SENDRECV(data[1, *], above, data[n*, *], below)`
- `MPI_SENDRECV(data[n, *], below, data[0, *], above)`

## 13 билет: Сети Петри: определение, примеры, вариации (сети с приоритетами, ингибиторные сети, цветные сети), моделирование параллельных процессов

### 1. Определение сети Петри

Сеть Петри – **двудольный ориентированный граф**, содержащий вершины двух типов:
- **места** (обозначаются кружками);
- **переходы** (обозначаются прямоугольниками или полосками).

Любая дуга ведёт либо от места к переходу, либо от перехода к месту. Дуги между двумя местами или двумя переходами запрещены.

**Формальное определение:** четвёрка `<S, T, W, μ₀>`, где
- `S` – конечное множество мест (`|S| = n`);
- `T` – конечное множество переходов (`|T| = m`, `S ∩ T = ∅`);
- `W : (S×T) ∪ (T×S) → Z₊` – мультимножество дуг;
- `μ₀ : S → Z₊` – **начальная разметка** (распределение меток по местам).

**Метки (токены, маркеры)** – все одинаковые, неотличимые.

### 2. Правила срабатывания

- **Переход активен**, если каждое его **входное место** содержит по крайней мере одну метку (по одной метке на каждую входящую дугу).
- **Активный переход может сработать**:
  - поглощает по одной метке с каждого входного места;
  - размещает по одной метке на каждое выходное место.
- В каждый момент времени из всех активных переходов **недетерминированно** выбирается один для срабатывания. Если активных переходов нет, работа сети завершается.

**Последовательность срабатываний** `σ` – последовательность символов переходов, сработавших на каждом шаге. Она однозначно определяет последовательность разметок `μᵢ`.

### 3. Свойства сети Петри

- **Безопасное место** – в любой достижимой разметке число меток в нём ≤ 1.  
  **Безопасная сеть** – все места безопасны.
- **Ограниченное место** – существует `K` такое, что в любой достижимой разметке число меток ≤ K.  
  **Ограниченная сеть** – все места ограничены.
- **Консервативная сеть** – число меток в любой достижимой разметке равно числу меток в начальной разметке (модель ресурсов, которые не создаются и не уничтожаются).

**Свойства сети как системы (проблемы анализа):**
- достижимость (возможность перехода из одного состояния в другое);
- ограниченность;
- безопасность (частный случай ограниченности, K=1).

### 4. Вариации (расширения) сетей Петри

#### 4.1. Сети Петри с приоритетами
Каждому переходу `t` ставится в соответствие число `Pr(t)` – приоритет.  
Если два перехода конфликтуют из‑за общего ресурса, преимущество получает переход с большим приоритетом. Если все приоритеты различны – конфликтов нет.

#### 4.2. Ингибиторные сети Петри
К стандартным дугам из мест в переходы добавляются **тормозящие дуги**.  
При наличии хотя бы одной метки в месте, связанном ингибиторной дугой, переход **не активируется** (даже если все входные места имеют метки). Переход активируется только тогда, когда в этом месте меток нет. Позволяет выполнять проверку на отсутствие меток.

#### 4.3. Цветные сети Петри
Каждая метка имеет **значение (цвет)** – простой (число, символ) или любой сложный тип.  
Переходы определяют отношения между значениями входных и выходных меток:  
- входным дугам приписываются **предусловия**, определяющие, метки с какими значениями поглощаются;
- выходные дуги определяют с помощью выражений, какие значения будут помещены в выходные места.

#### 4.4. Временные сети Петри
Метки получают атрибут **время задержки**.  
Если метка появилась в месте в момент `t` и имеет задержку `τ`, она будет доступна для переходов только начиная с момента `t+τ`. Задержки приписываются дугам, ведущим от переходов; все метки, поставляемые по такой дуге, получают одинаковую задержку.

### 5. Моделирование параллельных процессов

**Конфликт** – ситуация, когда несколько активных переходов претендуют на общую метку некоторого места.

**Схема параллельной обработки переходов:**
1. Из всех активных переходов выбирается **бесконфликтное подмножество** (никакие два не имеют взаимного конфликта).
2. Все выбранные переходы срабатывают **одновременно**.
3. Если активных переходов нет – сеть завершает работу.

**Принцип максимального параллелизма:** выбирается такое подмножество `A' ⊆ A` (множество всех активных переходов), что:
- никакие два перехода из `A'` не конфликтуют;
- для любого невыбранного перехода из `A \ A'` имеется хотя бы один конфликтующий переход в `A'` (набор не расширяем бесконфликтно). Наибольший по числу переходов не требуется, важно только, чтобы он был не расширяем.

**Пример моделирования** – система из двух светофоров, работающих согласованно: когда на первом горит красный, на втором – зелёный, и наоборот.

Сети Петри широко используются для моделирования:
- конвейерной обработки данных;
- критических секций / семафоров;
- механизмов синхронизации;
- параллельных и распределённых систем, а также естественных процессов (сети химических реакций).

## 14 билет: Генетические алгоритмы: операторы, кодирование, сходимость (теория схем), островная модель, клеточные ГА

### 1. Общие понятия

Генетические алгоритмы (ГА) – метод решения сложных (многомерных) оптимизационных задач, прежде всего дискретной оптимизации.

Рассматривается задача минимизации целевой функции `F(x₁,…,xₙ)`, `xᵢ ∈ {0,1}`. Любой набор из `n` значений – возможное решение.  
Целевая функция преобразуется в **функцию качества (fitness function)** `f(x) ≥ 0`, подлежащую максимизации.

**Генотип** – двоичный набор длины `n` (модель ДНК‑кода).  
**Гены** – отдельные биты.  
**Фенотип** – само решение, определяемое генотипом.

### 2. Итерации (операторы) ГА

Работа начинается с генерации **начальной популяции** (случайно). Размер популяции `m` обычно постоянен. Для каждого решения `xₖ` вычисляется `f(xₖ)`.

Каждая итерация (поколение) состоит из трёх операторов:

1. **Отбор (selection)** – удаление наименее приспособленных решений.
2. **Скрещивание (crossover)** – разбиение решений на пары; каждая пара с некоторой вероятностью обменивается частями генотипов.
3. **Мутация (mutation)** – для каждого решения с некоторой вероятностью производится незначительная случайная модификация генотипа.

Критерии останова: нахождение нужного решения, исчерпание числа поколений и т.п.

**Методы отбора:**
- **Рулетка** – вероятность выбора пропорциональна значению функции качества.
- **Ранговый отбор** – вероятность пропорциональна позиции в отсортированном списке, а не значению.
- **Турнирный метод** – из популяции случайно выбирается несколько особей, побеждает лучшая.

### 3. Способы кодирования и соответствующие операторы

#### 3.1. Двоичное кодирование (`xᵢ ∈ {0,1}`)
- **Скрещивание:** одноточечное, двухточечное, равномерное.
- **Мутация:** инверсия бита (`0→1`, `1→0`) с заданной вероятностью.

#### 3.2. Целочисленное кодирование (`xᵢ ∈ {a₁,…,aₙ}`)
- **Скрещивание:** одноточечное, двухточечное, равномерное.
- **Мутация:**  
  a) если параметры качественные (цвета, подграфы) – случайная замена на любое другое допустимое значение: `x ← Random(red, blue, green)` или `x ← Random(1..P)`.  
  b) если параметры количественные – увеличение или уменьшение на единицу (небольшое число): `xᵢ ← xᵢ + Random(-1, 1)`.

#### 3.3. Непрерывное кодирование (`xᵢ ∈ [a, b]`)
- **Скрещивание:** одноточечное, двухточечное, равномерное, **арифметическое** – значение каждого гена потомка вычисляется как среднее генов родителей:  
  `xᵢ' = α xᵢ + (1-α) yᵢ`, `yᵢ' = α yᵢ + (1-α) xᵢ`, где `α ∈ [0,1]`.
- **Мутация:** локальная или глобальная.

#### 3.4. Кодирование перестановками (задача коммивояжёра)
Обычные операторы скрещивания могут давать некорректные решения (не перестановки).  
Применяется **PMX (Partially Mapped Crossover)**:
1. Выбирается случайный сегмент `[k₁, k₂]`.
2. В потомок `x'` копируются элементы `x` из этого сегмента.
3. Для остальных позиций берутся элементы из `y`; если элемент уже есть в сегменте `x'`, он заменяется соответствующим элементом из `y` (с отображением).
4. Аналогично строится `y'`.

**Мутации перестановок:**
- **вставка** – два случайных элемента `xᵢ`, `xⱼ` (j>i), после чего `xᵢ` вставляется сразу за `xⱼ`;
- **обмен** – перестановка двух случайных элементов;
- **инверсия** – изменение порядка элементов между двумя случайными позициями на обратный.

### 4. Сходимость генетических алгоритмов. Теория схем

**Схема** – набор двоичных строк, в которых значения некоторых битов зафиксированы, а остальные обозначены `*`.  
Пример: `1*0` (порядок `o(h)=2`, определяющая длина `l(h)=2`) соответствует строкам `{100, 110}`.

**Порядок схемы** `o(h)` – количество фиксированных битов.  
**Определяющая длина** `l(h)` – расстояние между крайними фиксированными позициями.  
**Примеры схемы** – все строки, получаемые подстановкой 0/1 вместо `*`.

**Функция пригодности (ФП) схемы** – среднее значение ФП всех её примеров.

**Теорема схем** (неравенство):

\[
N(h, t+1) \ge N(h, t) \frac{f(h,t)}{\bar{f}(t)} (1 - p)
\]

где
- `N(h,t)` – количество примеров схемы `h` на шаге `t`;
- `f(h,t)` – ФП схемы;
- `\bar{f}(t)` – средняя ФП по популяции;
- `p` – вероятность уничтожения схемы под действием операторов:

\[
p = \frac{l(h)}{n-1} p_c + o(h) p_m
\]

`p_c` – вероятность скрещивания, `p_m` – вероятность мутации.

**Интерпретация:**  
Хорошо приспособленные схемы с **малым порядком** и **малой определяющей длиной** (строительные блоки) экспоненциально распространяются в популяции.  
- Мутации реже разрушают схемы меньшего порядка.  
- Скрещивание реже разрушает схемы с меньшей определяющей длиной.

### 5. Параллельные модели генетических алгоритмов

#### 5.1. Островная модель
- Вся популяция разбивается на субпопуляции, каждая помещается на отдельный процессор.
- Субпопуляции развиваются независимо по обычным правилам ГА.
- Добавляется оператор **миграция** – после нескольких шагов ГА производится обмен частью решений с соседними субпопуляциями (процессорами, имеющими прямую связь).
- Варьируя размер отбираемой части и частоту, можно управлять степенью миграции (от высокой до низкой).  
- Преимущество: естественное отображение на архитектуру межпроцессорных связей; возможность использовать любые операторы отбора.

#### 5.2. Клеточные генетические алгоритмы (микроскопическая модель)
- Все особи помещаются в отдельные клетки (одно решение – одна клетка) клеточной структуры (чаще всего прямоугольная 2D или 3D решётка). Каждая клетка имеет небольшое фиксированное число соседей.
- **Бинарные операторы** применяются только к решениям, расположенным в соседних клетках.
- Отбор обычно турнирный; клетки разбиваются на пары соседей, каждая пара борется за выживание.
- Скрещивание – пара соседних решений с некоторой вероятностью скрещивается, новые решения записываются вместо родителей.
- Миграция также реализуется между соседними клетками.

Клеточная модель – «мягкий» вариант островной; образуются устойчивые субпопуляции без явных границ, возникает конкурентная борьба между ними.

#### 5.3. Неоднородные клеточные ГА
Некоторые параметры ГА (например, вероятность мутации `d_{mut}`) делаются зависимыми от положения особи в клеточном пространстве.  
Области с высокой мутацией постоянно генерируют случайные решения, поддерживая разнообразие популяции; области с низкой мутацией используются для селекции лучших решений.  
Пример: градиент величины мутации – от краёв (высокая) к центру (низкая).

## 15 билет: Информационная безопасность. Шифрование данных. Криптографическая стойкость. Симметричная криптография. Блочный шифр (DES) и его режимы. Асимметричные схемы (RSA и Диффи‑Хеллмана). Код аутентификации (MAC). Цифровая подпись (DSA).

### 1. Информационная безопасность

По ГОСТ Р 50922‑2006: **информационная безопасность** – состояние защищённости информации (данных), при котором обеспечены её конфиденциальность, целостность и доступность.

- **Конфиденциальность** – доступ к информации только авторизованным пользователям.
- **Целостность** – достоверность и полнота информации и методов её обработки.
- **Доступность** – доступ к информации и связанным активам авторизованных пользователей по мере необходимости.

### 2. Шифрование данных

**Шифрование** – обратимое преобразование информации для сокрытия от неавторизованных лиц с предоставлением доступа авторизованным пользователям.  
Важная особенность – использование **ключа**, который выбирает конкретное преобразование из совокупности возможных.

**Составляющие:** зашифрование и расшифрование.  

**Обеспечиваемые состояния безопасности:**
1. **Конфиденциальность** – скрытие информации при передаче или хранении.
2. **Целостность** – предотвращение изменения информации.
3. **Идентифицируемость** – аутентификация источника и предотвращение отказа от авторства.

### 3. Криптографическая стойкость

Способность криптографического алгоритма противостоять криптоанализу.  
Алгоритм считается стойким, если успешная атака требует недостижимых вычислительных ресурсов или перехваченных сообщений, либо время раскрытия таково, что информация к тому моменту утратит актуальность.  
В большинстве случаев криптостойкость не может быть математически доказана; можно лишь доказать уязвимости или свести задачу вскрытия к вычислительно сложной проблеме.

### 4. Симметричная криптография

Один и тот же ключ используется для шифрования и расшифрования. Ключ должен сохраняться в секрете обеими сторонами. До появления асимметричных схем был единственным способом.

### 5. Блочный шифр DES (Data Encryption Standard)

Разработан IBM, утверждён как стандарт США в 1977 (FIPS 46‑3).  
**Параметры:**
- Размер блока – 64 бита.
- Длина ключа – 56 бит.
- **Сеть Фейстеля** с 16 раундами.
- Комбинация нелинейных (S‑блоки) и линейных (перестановки E, IP, IP⁻¹) преобразований.

**Режимы DES:**
- **ECB (Electronic Code Book)** – режим электронной кодовой книги (простая замена).
- **CBC (Cipher Block Chaining)** – режим сцепления блоков.
- **CFB (Cipher FeedBack)** – режим обратной связи по шифротексту.
- **OFB (Output FeedBack)** – режим обратной связи по выходу.
- **Counter Mode (CM)** – режим счётчика.

**Развитие:** **Triple DES (3DES)** – троекратное выполнение DES (шифрование/расшифровка).

### 6. Асимметричные схемы

Криптосистема с открытым ключом: открытый ключ передаётся по открытому каналу (для шифрования и проверки подписи), закрытый ключ (для расшифровки и создания подписи) держится в секрете. Широко применяется в TLS/SSL, SSH.

#### 6.1. RSA

**Генерация ключей:**
1. Выбрать два различных случайных простых числа `p` и `q` (например, 1024 бита каждое).
2. Вычислить `n = p·q` (модуль).
3. Вычислить функцию Эйлера `φ(n) = (p-1)(q-1)`.
4. Выбрать открытую экспоненту `e` (`1 < e < φ(n)`, взаимно простое с `φ(n)`). Обычно берут простые числа Ферма: 17, 257, 65537 (для ускорения возведения в степень).
5. Вычислить секретную экспоненту `d`, обратную к `e` по модулю `φ(n)`: `d·e ≡ 1 (mod φ(n))` (расширенный алгоритм Евклида).
6. Открытый ключ: `(e, n)`. Закрытый ключ: `(d, n)`.

**Шифрование (Бобом для Алисы):**
`c = E(m) = m^e mod n`

**Расшифрование (Алисой):**
`m = D(c) = c^d mod n`

#### 6.2. Диффи‑Хеллман (DH)

Протокол для получения общего секретного ключа по незащищённому каналу, используемого затем для симметричного шифрования.  
Уязвим для атаки «человек посередине» без дополнительной аутентификации.

**Для двух участников:**
1. Стороны договариваются о параметрах `p` (простое) и `g` (первообразный корень).
2. Алиса генерирует секретное `a`, вычисляет `A = g^a mod p` и отправляет Бобу.
3. Боб генерирует `b`, вычисляет `B = g^b mod p` и отправляет Алисе.
4. Алиса вычисляет `K = B^a mod p = g^{ab} mod p`.
5. Боб вычисляет `K = A^b mod p = g^{ab} mod p`.

**Для трёх участников (Алиса, Боб, Кэрол):**
1. Договариваются о `p`, `g`.
2. Генерируют секретные `a`, `b`, `c`.
3. Алиса → Боб: `g^a`.  
   Боб → Кэрол: `(g^a)^b = g^{ab}`.  
   Кэрол → (Алиса?) – в материале описан полный протокол:  
   - Боб → Кэрол: `g^b`.  
   - Кэрол → Алиса: `(g^b)^c = g^{bc}`.  
   - Алиса → Боб: `(g^c)^a = g^{ca}`.  
   - Затем каждый вычисляет `g^{abc}`:  
     Кэрол: `(g^{ab})^c = g^{abc}`;  
     Алиса: `(g^{bc})^a = g^{abc}`;  
     Боб: `(g^{ca})^b = g^{abc}`.

### 7. Код аутентификации (MAC)

**MAC (Message Authentication Code)** – имитовставка, добавляемая к сообщению для обеспечения целостности и аутентификации источника данных.  
Некоторые хэш‑функции (например, MD5) не могут использоваться в MAC напрямую, так как принимают только один аргумент (данные) и не используют секретный ключ.

**HMAC (Hash‑based MAC)** – механизм с секретным ключом и двукратным применением хеш‑функции `H`. Виды: HMAC‑MD5, HMAC‑SHA1, HMAC‑RIPEMD128, HMAC‑RIPEMD160 и др.

Формула HMAC:

\[
\text{HMAC}_K(\text{text}) = H\big((K \oplus \text{opad}) \| H((K \oplus \text{ipad}) \| \text{text})\big)
\]

где
- `ipad` – блок из `b` байт 0x36 (inner);
- `opad` – блок из `b` байт 0x5c (outer);
- `K` – ключ;
- `⊕` – XOR;
- `‖` – конкатенация строк (байтов);
- `b` – размер блока хеш‑функции.

### 8. Цифровая подпись DSA (Digital Signature Algorithm)

Криптографический алгоритм с закрытым ключом для создания электронной подписи (не для шифрования). Основан на вычислительной сложности логарифмирования в конечных полях.

**Генерация параметров и ключей:**
1. Выбрать криптографическую хеш‑функцию `H(x)`, размер выхода `N` бит.
2. Выбрать простое число `q` размерностью `N` бит.
3. Выбрать простое число `p` размерностью `L` бит, такое что `(p-1)` делится на `q`.
4. Выбрать `g` (порядок `q` по модулю `p`): `g = h^{(p-1)/q} mod p`, где `h` – произвольное число, `1 < h < p-1`, `g ≠ 1` (часто `h=2`).
5. **Секретный ключ:** случайное число `x`, `0 < x < q`.
6. **Открытый ключ:** `y = g^x mod p`.

**Подпись сообщения `m`:**
1. Выбрать случайное `k`, `0 < k < q`.
2. Вычислить `r = (g^k mod p) mod q`. Если `r=0` – выбрать другое `k`.
3. Вычислить `s = k^{-1} (H(m) + x·r) mod q`. Если `s=0` – выбрать другое `k`.
4. Подпись – пара `(r, s)` (общая длина `2N` бит).

**Проверка подписи:**
1. Вычислить `w = s^{-1} mod q`.
2. Вычислить `u₁ = H(m)·w mod q`, `u₂ = r·w mod q`.
3. Вычислить `v = (g^{u₁}·y^{u₂} mod p) mod q`.
4. Подпись верна, если `v = r`.

## 16 билет: Односторонняя и двусторонняя модели передачи сообщений в библиотеке MPI, принципы программирования и синхронизации параллельных процессов. Протоколы Eager и Rendezvous для реализации двусторонней модели.

### 1. Двусторонняя модель (point‑to‑point)

Оба процесса (отправитель и получатель) явно участвуют в коммуникации.  
Операции могут быть:
- **блокирующие** – `MPI_Send`, `MPI_Recv`;
- **неблокирующие** – `MPI_Isend`, `MPI_Irecv` (возвращают управление сразу, завершение проверяется `MPI_Wait` или `MPI_Test`);
- **персистентные** – полезны при многократном повторении одного и того же шаблона обмена.  
  Инициализация: `MPI_Send_init`, `MPI_Recv_init` – создают объект запроса. Затем в цикле: `MPI_Start` (или `MPI_Startall`), `MPI_Wait`. Завершение: `MPI_Request_free`.

**Режимы передачи сообщений:**
- **Стандартный режим (Standard)** – реализация сама решает, буферизовать сообщение или нет.  
  a) сообщение помещается в буфер, вызов `MPI_Send` завершается до вызова соответствующего `MPI_Recv` (локальный).  
  b) буфер недоступен – вызов не завершится, пока не будет вызван соответствующий `MPI_Recv` (нелокальный).
- **Режим с буферизацией (Buffered)** – `MPI_Bsend` завершается независимо от вызова `MPI_Recv`, сообщение помещается в заранее выделенный буфер (локальный).
- **Синхронный режим (Synchronous)** – `MPI_Ssend` завершается, только когда соответствующий `MPI_Recv` начал приём сообщения (нелокальный + синхронизация).
- **Режим по готовности (Ready)** – `MPI_Rsend` может начать передачу, если соответствующий `MPI_Recv` уже вызван; позволяет избежать «рукопожатия» для сокращения времени (локальный).

### 2. Протоколы Eager и Rendezvous для двусторонней модели

#### 2.1. Eager (жаждущий) протокол
Отправитель полагает, что каждое сообщение, любого размера, может быть принято получателем.  
- Сообщение отправляется немедленно, даже если получатель ещё не вызвал `MPI_Recv`.  
- Получатель буферизует пришедшие данные до вызова `MPI_Recv`.  
**Свойства:**  
- минимизирует время на синхронизацию;  
- может тратить лишние буферы;  
- требует активного участия CPU получателя для обработки входящего потока;  
- минимизирует латентность за счёт перекрытия;  
- может вызвать перегрузку сети.

#### 2.2. Rendezvous протокол
Пересылка начинается только после того, как получатель сигнализирует, что его буфер готов.  
- Сначала передаются метаданные сообщения.  
- Получатель, получив метаданные, подтверждает готовность, после чего идёт передача данных.  
**Свойства:**  
- требует меньше буферов;  
- добавляет задержку на «рукопожатие»;  
- безопаснее для больших сообщений.

Выбор протокола обычно зависит от размера сообщения: для маленьких сообщений часто используется Eager, для больших – Rendezvous. Точные пороги определяются реализацией MPI.

### 3. Односторонняя модель (RMA – Remote Memory Access)

Данные перемещаются из памяти одного процесса в память другого, при этом в передаче явно участвует только один процесс (инициатор). Второй процесс может вообще не вызывать MPI‑функций, связанных с обменом. Модель близка к разделяемой памяти.

**Окно (window)** – область памяти, выделенная для односторонних операций. Создание окна – коллективная операция (`MPI_Win_create`).  
Пример создания окна (в псевдокоде):

    int window_buffer[4] = {0};
    if (rank == 1) {
        window_buffer[0] = 42;  window_buffer[1] = 88;
        window_buffer[2] = 12;  window_buffer[3] = 3;
    }
    MPI_Win win;
    MPI_Win_create(&window_buffer, (MPI_Aint)4*sizeof(int), sizeof(int),
                   MPI_INFO_NULL, comm, &win);

**Активная синхронизация** – оба процесса участвуют в синхронизации, явно вызывая функции:
- `MPI_Win_fence` – барьерная синхронизация;
- `MPI_Win_post` / `MPI_Win_start` / `MPI_Win_complete` / `MPI_Win_wait` – групповой вариант.

**Пассивная синхронизация** – только инициатор управляет синхронизацией; целевой процесс не вызывает функций, но доступы сериализуются с помощью мьютекса, который инициатор блокирует.  
Функции:
- `MPI_Win_lock` – блокировка окна (например, `MPI_LOCK_EXCLUSIVE`);
- `MPI_Win_unlock` – снятие блокировки.

**Операции односторонней передачи:**
- `MPI_Put` – запись в удалённую память;
- `MPI_Get` – чтение из удалённой памяти;
- `MPI_Accumulate` – атомарное обновление.

**Пример (фрагмент из материала):**

    int getbuf[4];
    if (rank == 0) {
        MPI_Get(&getbuf, 4, MPI_INT, 1, 0, 4, MPI_INT, win);
    }
    MPI_Win_fence(0, win);

Здесь процесс 0 читает данные из окна процесса 1. Обрамляющий `MPI_Win_fence` (активная синхронизация) ограничивает **эпоху доступа**.

**Эпоха** – набор операций между синхронизациями. В односторонней модели различают эпохи доступа (access epoch) и эпохи воздействия (exposure epoch). Для пассивной синхронизации эпоха начинается с `MPI_Win_lock` и заканчивается `MPI_Win_unlock`.

## 17 билет: Параллельные алгоритмы обработки графов. Алгоритмы поиска вширь, Беллмана-Форда, дельта-степпинга, Борувки, лувенский, распространения меток, Брандеса.

### 1. Общие проблемы анализа больших графов

Перед рассмотрением конкретных алгоритмов важно отметить фундаментальные трудности, которые возникают при параллельной обработке больших графов на суперкомпьютерах (по материалу лекций):

- **Data‑driven computations** – зависимость вычислений от данных (топологии графа); невозможность применения методов статического распараллеливания.
- **Unstructured problems** – работа с нерегулярными, неструктурированными данными, трудность распараллеливания.
- **Poor locality** – низкая пространственно‑временная локализация обращений к памяти (доступ к данным разбросан, кэш‑промахи).
- **High data access to computation ratio** – преобладание команд доступа к памяти над командами арифметических операций.

Эти особенности ограничивают эффективность параллельных графовых алгоритмов и заставляют разрабатывать специальные методы (декомпозиция, переупорядочивание, асинхронные обмены и т.д.).

### 2. Алгоритм поиска в ширину (BFS)

**Последовательный BFS** (классический) – обход графа от исходной вершины `s` с использованием очереди. Для каждой вершины хранится расстояние (количество рёбер) от `s` и предшественник.

**Параллельный BFS** на распределённой памяти:
- Граф разбивается на домены (каждый процессор получает подмножество вершин и инцидентные им рёбра).
- Выполняется волновой алгоритм: на каждой итерации (уровне BFS) каждый процесс проверяет своих локальных вершин, которые ещё не посещены и имеют соседа на предыдущем уровне.
- Возникает необходимость обмена «гало»‑вершинами (граничные вершины, чьи соседи находятся на другом процессоре).
- Для минимизации обменов часто применяется **2D‑декомпозиция** (разбиение матрицы смежности на блоки) или метод **выделения вершин‑фронта**.
- Оценка производительности: BFS на графах с нерегулярной структурой страдает от низкой локальности и большого числа сообщений MPI (особенно при плохой кластеризации).

В материалах лекций BFS не описан подробно, но он является основой для многих алгоритмов (например, алгоритма Брандеса для вычисления посредничества).

### 3. Алгоритм Беллмана‑Форда (Bellman‑Ford)

**Назначение:** поиск кратчайших путей от одной вершины до всех остальных во взвешенном графе (допускаются рёбра с отрицательным весом). Время работы – `O(|V|·|E|)`.

**Шаги алгоритма (последовательная версия):**

1. **Инициализация:** массив расстояний `dist[]` размера `|V|` заполняется бесконечностью, `dist[src] = 0`.
2. **Основной цикл** (повторяется `|V|‑1` раз):
   - Для каждого ребра `(u, v)` с весом `w`:
     - Если `dist[v] > dist[u] + w`, то `dist[v] = dist[u] + w`.
3. **Проверка на циклы отрицательного веса:**
   - Для каждого ребра `(u, v)`:
     - Если `dist[v] > dist[u] + w`, то в графе присутствует цикл отрицательного веса (и задача не имеет корректного решения).

**Параллельная реализация** (MPI):
- Граф распределяется между процессами (по вершинам или по рёбрам).
- На каждой итерации каждый процесс обновляет расстояния для своих локальных вершин и рёбер.
- После каждого `|V|‑1` шага требуется глобальная синхронизация (например, `MPI_Allreduce` для проверки изменений) или асинхронная модель с помощью односторонних коммуникаций.
- Для графов с отрицательными весами параллельный Беллман‑Форд остаётся сложным из‑за зависимостей по данным.

### 4. Алгоритм дельта‑степпинга (Delta‑Stepping)

**Назначение:** параллельный алгоритм для задачи поиска кратчайших путей из одного источника (SSSP) на графах с неотрицательными весами. Является усовершенствованием волнового подхода.

**Основная идея:** рёбра группируются по весу с использованием параметра `Δ` (дельта). Вершины обрабатываются в «вёдрах» (buckets), каждое ведро соответствует диапазону расстояний шириной `Δ`.

**Псевдокод (по лекциям):**

```
DeltaStepping(G, s, delta)
    i = 0
    while true
        processBucket(i)
        i = min(k > i : Bk != {})
        if i == inf then break
```

- Начальное расстояние до `s` равно 0, все остальные – бесконечность.
- Вёдра `B[i]` содержат вершины, чьи текущие расстояния лежат в интервале `[i·Δ, (i+1)·Δ)`.
- На каждом шаге обрабатывается текущее ведро (вершины из него извлекаются, для каждой рассматриваются исходящие рёбра, для соседей расстояния потенциально уменьшаются, и они могут переместиться в более раннее ведро).
- Для неотрицательных весов, равномерно распределённых в `[0,1]`, рекомендуется выбирать `Δ = 1/d`, где `d` – максимальная степень вершины.

**Сложность:** `O(M log N)` для последовательной версии. Количество итераций параллельного алгоритма оценивается как `O( (d_c / Δ) * (log n / log log n) )`, где `d_c` – максимальная длина пути.

**Параллелизм:** ведро может обрабатываться параллельно, если для каждого ребра нет конфликтов записи в расстояния соседей. Используются техники раскраски графа или атомарные обновления.

### 5. Алгоритм Борувки (Borůvka’s algorithm)

**Назначение:** построение минимального остовного дерева (MST) во взвешенном неориентированном связном графе. Впервые опубликован в 1926 году.

**Алгоритм:**

1. Изначально каждая вершина – тривиальное дерево, рёбра не принадлежат никакому дереву.
2. Пока количество деревьев больше 1:
   - Для каждого дерева `T` найти минимальное по весу ребро, инцидентное `T` (одно из рёбер, соединяющих `T` с другим деревом).
   - Добавить все найденные рёбра в MST (соединить деревья).
3. Повторять шаг 2 до тех пор, пока не останется одно дерево.

**Особенность:** если в графе есть рёбра одинакового веса, алгоритм может добавить лишние рёбра (образовать цикл). Для избежания этого при выборе минимального ребра используют вторичный критерий (например, наименьший номер ребра).

**Асимптотика:** на каждой итерации число компонент уменьшается как минимум в 2 раза, поэтому внешний цикл выполняется `O(log |V|)` раз. Внутренний цикл (поиск минимального ребра для каждой компоненты) требует `O(|E|)` операций на итерацию. Итоговая сложность `O(|E| log |V|)`.

**Параллельная реализация:**
- Каждый процессор получает подграф (набор вершин и рёбер).
- Внутри своего подграфа каждый процесс находит минимальные исходящие рёбра для своих деревьев.
- Затем выполняется глобальная редукция (например, `MPI_Allreduce`) для определения глобально минимальных рёбер на каждой итерации.
- Благодаря логарифмическому числу итераций алгоритм хорошо масштабируется.

### 6. Лувенский алгоритм (Louvain method)

**Назначение:** обнаружение сообществ (community detection) в неориентированных взвешенных графах. Сообщество – множество вершин, внутренние связи которого сильнее, чем внешние.

**Ключевое понятие – модулярность (modularity):**
- Модулярность равна доле рёбер от общего числа рёбер, которые попадают в данные группы, минус ожидаемая доля рёбер, которые попали бы в те же группы при случайном распределении.
- Значение модулярности лежит в интервале `[-1, 1]`. Положительная модулярность означает, что число рёбер внутри групп превышает случайное ожидание.

**Алгоритм (итеративный, две фазы):**

1. **Локальная оптимизация модулярности:**
   - Каждая вершина помещается в своё собственное сообщество.
   - Последовательно перебираются вершины; для каждой вершины рассматривается перемещение в сообщество каждого из её соседей. Вычисляется прирост модулярности `ΔQ`.
   - Вершина помещается в то сообщество, где `ΔQ` максимален и положителен. Если положительного прироста нет, вершина остаётся в своём сообществе.
   - Процесс повторяется до тех пор, пока ни одно перемещение не увеличивает модулярность.

2. **Агрегация:**
   - Каждое полученное сообщество сворачивается в одну «супервершину».
   - Внутренние рёбра сообщества становятся петлями (с суммарным весом), рёбра между сообществами – рёбрами между супервершинами (вес суммируется).
   - Получается взвешенный граф меньшего размера.

Затем фазы 1 и 2 повторяются на сжатом графе до тех пор, пока модулярность перестаёт расти.

**Свойства:**
- Быстрый (почти линейное время на разреженных графах).
- Хорошо параллелится: фаза 1 может выполняться на разных вершинах независимо (с осторожностью при конфликтах), фаза 2 – свёртка сообществ – также параллельна.
- Результат зависит от порядка обхода вершин (обычно используют случайный порядок на каждой итерации).

### 7. Алгоритм распространения метки (Label Propagation)

**Назначение:** быстрое обнаружение сообществ (почти линейное время `O(|E|)`).

**Идея:** каждая вершина присоединяется к тому сообществу, которому принадлежит большинство её соседей. Если таких сообществ несколько – выбирается случайное.

**Алгоритм (по материалу лекций):**

```
component_id[u] = u, u ∈ V
do
    updated = false
    for each u ∈ V (в случайном порядке)
        l = argmax_c [ Σ_{e(u,v)∈E, v∈c} w(e) ]   # сообщество c, дающее максимальную сумму весов рёбер к соседям из c
        if component_id[u] ≠ l
            component_id[u] = l
            updated = true
        end if
    end for
while updated
```

- В начальный момент времени каждой вершине присваивается уникальная метка (своё сообщество).
- На каждой итерации вершины обрабатываются в случайном порядке: для каждой вычисляется, какая метка даёт наибольший суммарный вес инцидентных рёбер, ведущих к вершинам с этой меткой.
- Вершина принимает эту метку (если она отличается от текущей).
- Алгоритм завершается, когда ни одна вершина не меняет метку.

**Особенности:**
- Количество итераций слабо зависит от размера графа (обычно 5–10).
- Нет уникального решения (случайность влияет на результат).
- Иногда может давать плохое решение – одно большое сообщество (все вершины сливаются).
- Сложность `O(|E|)`. Параллелизм достигается за счёт независимой обработки вершин (но требуется аккуратность при записи меток – можно использовать конвейер или расщепление итераций).

### 8. Алгоритм Брандеса (Brandes’ algorithm)

**Назначение:** вычисление **центральности по посредничеству (betweenness centrality)** для всех вершин графа. Показывает, насколько часто вершина лежит на кратчайших путях между другими вершинами.

**Определения:**
- `σ_{st}` – количество кратчайших путей от `s` до `t`.
- `σ_{st}(v)` – количество таких путей, проходящих через `v`.
- Центральность по посредничеству вершины `v`:
  \[
  BC(v) = \sum_{s \neq v \neq t} \frac{\sigma_{st}(v)}{\sigma_{st}}
  \]
- Зависимость (dependency) вершины `s` на вершине `v`: 
  \[
  \delta_{s}(v) = \sum_{t : v \in P_{st}} \frac{\sigma_{st}(v)}{\sigma_{st}}
  \]
  где `P_{st}` – множество кратчайших путей. Можно показать, что `BC(v) = \sum_{s} \delta_{s}(v)`.

**Алгоритм Брандеса (последовательный) для невзвешенного графа:**

Для каждой исходной вершины `s` выполняются три этапа:

1. **Поиск кратчайших путей (BFS):**
   - Инициализировать: `d[s] = 0`, `σ[s] = 1`, очередь `Q` c `s`.
   - Пока `Q` не пуста:
     - Извлечь `v`, положить в стек `S`.
     - Для каждого соседа `w`:
       - Если `d[w] == -1` (не посещён): `d[w] = d[v] + 1`, поместить `w` в `Q`.
       - Если `d[w] == d[v] + 1`: `σ[w] += σ[v]` и добавить `v` в список предшественников `P[w]`.

2. **Накопление зависимостей (обратный проход):**
   - Инициализировать `δ[v] = 0` для всех `v`.
   - Пока `S` не пуст:
     - Извлечь `w` из `S`.
     - Для каждого `v ∈ P[w]`:
       - `δ[v] += (σ[v] / σ[w]) * (1 + δ[w])`.
     - Если `w != s`: `BC[w] += δ[w]`.

3. **Суммирование** по всем `s` в `BC[v]`.

**Сложность:** `O(|V|·|E|)` для невзвешенных графов, `O(|V|·|E| + |V|^2·log|V|)` для взвешенных (с использованием алгоритма Дейкстры вместо BFS). Объём памяти `O(|V|+|E|)`.

**Параллельная реализация (MPI):**
- Внешний цикл по `s` можно распараллелить: разные процессы обрабатывают разные исходные вершины.
- После завершения своей части каждый процесс частично накапливает `BC[v]`; затем выполняется глобальная редукция (суммирование).
- Для больших графов используют также распределение данных (матрицы смежности) и асинхронные BFS.

**Псевдокод из лекций (BC‑Brandes):**

```
BC‑Brandes(G) // G – связный неориентированный невзвешенный граф
for all v in V do BC[v] = 0
for all s in V do
    BrandesInitS(s)
    ShortestPathsCounting(s)
    DependencyAccumulation()
end for
```

Функции:
- `BrandesInitS(s)`: инициализирует пустой стек `S`, списки предшественников `P[t]`, `σ[t]=0`, `d[t]=-1`, `δ[t]=0`. Устанавливает `σ[s]=1`, `d[s]=0`.
- `ShortestPathsCounting(s)`: BFS с очередью (как описано выше).
- `DependencyAccumulation()`: обратный проход по стеку `S` и накопление зависимостей.

**Применение:** оценка важности узлов в социальных сетях, транспортных сетях, сетях передачи данных.

## 18 билет: Математические основы GraphBLAS: данные, операции

**GraphBLAS** – стандарт проектирования графовых алгоритмов на языке линейной алгебры. Граф представляется в виде **разреженной матрицы** (обычно матрица смежности). Операции обхода графа моделируются через умножение матрицы на вектор и другие матричные операции.

### 1. Основные данные

- **Матрица** – разреженная, может храниться в формате CSR, CSC или других.
- **Вектор** – разреженный или плотный.
- **Скаляр** – элемент полукольца.
- **Маска** – битовая или числовая матрица/вектор, определяющая, какие элементы выходной матрицы могут быть изменены (ненулевые элементы маски разрешают запись).

### 2. Основные операции (по стандарту)

Для каждой операции, кроме `mxv` и `vxm`, предусмотрен как матричный, так и векторный варианты.

- **`mxv`** и **`vxm`** – умножение матрицы на вектор (или вектора на матрицу). Является ключевой операцией: она соответствует распространению информации от множества вершин по рёбрам графа.
- **`mtxm`** – умножение матрицы на матрицу (композиция отношений).
- **`eWiseMult`**, **`eWiseAdd`** – поэлементное умножение или сложение компонент двух матриц/векторов (по аналогии с поэлементными операциями над матрицами смежности).
- **`extract`** – извлечение подматрицы или подвектора по заданным индексам.
- **`assign`** – присваивание значений элементам матрицы/вектора по индексам (с учётом маски).
- **`apply`** – применение унарной функции к каждому элементу матрицы/вектора (например, `apply(log)` или `apply(not)`).
- **`reduce`** – редукция матрицы до одного скаляра (сумма всех элементов, произведение и т.д.) или до вектора (суммирование по строкам или столбцам).
- **`transpose`** – транспонирование матрицы.

### 3. Полукольца (semirings)

Универсальность GraphBLAS достигается за счёт возможности замены арифметических операций `+` и `*` на любые другие бинарные операции, удовлетворяющие свойствам полукольца. Полукольцо – это множество `S` с двумя операциями `⊕` (сложение) и `⊗` (умножение), такими что:
- `⊕` – ассоциативна, коммутативна, имеет нейтральный элемент `0`.
- `⊗` – ассоциативна, имеет нейтральный элемент `1`.
- `⊗` дистрибутивно относительно `⊕` слева и справа.
- Умножение на `0` даёт `0`: `a ⊗ 0 = 0 ⊗ a = 0`.

**Свойства, требуемые от операций в GraphBLAS:**
- Коммутативность `⊕` (не всегда требуется, но желательно).
- Ассоциативность `⊕` и `⊗`.
- Дистрибутивность `⊗` относительно `⊕`.
- Существование нейтрального элемента для `⊕` (обозначается `0`).
- `a ⊗ 0 = 0 ⊗ a = 0`.

**Примеры полуколец:**

| Полукольцо            | `⊕` (сложение) | `⊗` (умножение) | Интерпретация                     |
|-----------------------|----------------|-----------------|------------------------------------|
| Арифметическое        | `+`            | `*`             | Число путей, сумма весов          |
| Булево                | `OR` (∨)       | `AND` (∧)       | Существует ли путь?               |
| Min‑Plus (тропическое) | `min`          | `+`             | Длина кратчайшего пути            |
| Max‑Plus              | `max`          | `+`             | Самый длинный путь                |
| Max‑Min               | `max`          | `min`           | Пропускная способность            |

**Пример использования:** умножение вектора `v` (размерность `n`) на матрицу `A` (размер `n×n`) в полукольце `(min, +)` даёт вектор расстояний: `(v ⊗ A)[j] = min_i (v[i] + A[i][j])`. Это один шаг релаксации в алгоритме кратчайших путей.

### 4. Маски (Mask)

Маска – это матрица (или вектор) того же размера, что и результат операции. Её ненулевые элементы указывают, какие позиции результата могут быть изменены. Нулевые элементы маски блокируют запись (сохраняют предыдущее значение). Маски позволяют реализовать пошаговые алгоритмы, где нужно изменять только часть вершин (например, фронт BFS).

**Пример:** маска `Mask` – вектор, где `Mask[i]=1` если вершина `i` ещё не посещена. Тогда операция `v' = (v ⊗ A) .* Mask` (поэлементное умножение с маской) обновит расстояния только для непосещённых вершин.

### 5. Пример: алгоритм Беллмана‑Форда через GraphBLAS

В лекциях приводится, что алгоритм Беллмана‑Форда может быть выражен как последовательность операций `v = (v ⊗ A)` в полукольце `(min, +)` с использованием маски для контроля изменений. Граф представляется матрицей весов `W` (где `W[i][j] = weight(i,j)`, если ребро существует, иначе `∞`). Начальный вектор расстояний `d` содержит `0` для источника и `∞` для остальных. Повторяем `|V|-1` раз: `d = (d ⊗ W)` (в смысле `min‑plus` умножения). Для обнаружения отрицательных циклов делается ещё одна итерация и сравниваются изменения.

Таким образом, GraphBLAS предоставляет единый алгебраический язык для описания графовых алгоритмов, позволяя легко переключаться между разными семантиками (взвешенной, булевой, тропической) и эффективно реализовывать их на параллельных архитектурах (включая GPU и распределённые системы).

## 19 билет: Позиционные игры с полной информацией и игры общего вида. Смешанные стратегии и стратегии поведения. Алгоритм Куна определения совершенного подыгрового равновесия.

### 1. Позиционные игры с полной информацией

**Позиционная игра** моделируется конечным **деревом игры** – ориентированным графом без циклов, где вершины – позиции, дуги – возможные переходы. Корень `x₀` – начальная позиция. Вершины без преемников – **финальные** (множество `T`). Каждой финальной вершине приписаны выигрыши игроков.

**Формальное определение:** позиционная игра с полной информацией задаётся:
- Множеством игроков `A`.
- Деревом `(X, σ)`, где `σ: X → X` – отображение, ставящее в соответствие вершине её ближайшего предшественника (`σ(x₀) = x₀`).
- Разбиением нефинальных вершин: `Xᵃ` – личные позиции игрока `a`, `X⁰` – позиции случая (природы).
- Для позиций случая `x ∈ X⁰` заданы вероятности переходов `p(x' | x) > 0`, `∑_{x'} p(x'|x) = 1`.
- Функциями выигрыша `uᵃ: T → ℝ` для каждого игрока `a`.

Игрок в своей личной позиции `x ∈ Xᵃ` выбирает следующую вершину из множества альтернатив `σ⁻¹(x)`. Игра развивается от корня к финальной вершине.

**Чистая стратегия** игрока `a` – отображение `µᵃ`, которое каждой его личной позиции `x ∈ Xᵃ` ставит в соответствие одну из альтернатив `µᵃ(x) ∈ σ⁻¹(x)`. Набор стратегий всех игроков `µ = (µᵃ)_{a∈A}` называется **ситуацией**. Ситуация определяет вероятностное распределение на множестве финальных вершин (за счёт ходов случая) и, следовательно, ожидаемые выигрыши `uᵃ(µ)`.

**Нормальная форма** позиционной игры – это бескоалиционная игра, в которой чистыми стратегиями являются описанные выше отображения.

### 2. Подыгры и совершенное подыгровое равновесие (СПР)

**Подыгра** `G_z`, начинающаяся в вершине `z`, определяется как игра с деревом `X_z` – множеством вершин, достижимых из `z`, и сохранением всех остальных элементов (разбиения, вероятностей, выигрышей).

**Совершенное подыгровое равновесие** – такая ситуация `µ̂`, что для любой вершины `z ∈ X` её сужение `µ̂_z` на подыгру `G_z` является равновесием по Нэшу в `G_z`. Иными словами, стратегии оптимальны в любой возможной подыгре, даже той, которая в равновесии не достигается.

### 3. Алгоритм Куна (обратная индукция) нахождения СПР

Алгоритм работает для конечных позиционных игр с полной информацией.

1. Рассматриваются **префинальные вершины** `Z₁ = { x | σ⁻¹(x) ⊆ T }` (все преемники – финальные).
   - Если `x ∈ Xᵃ` (ходит игрок `a`), выбирается наилучшая для `a` альтернатива:
     \[
     \hat{p}^a(x) \in \arg\max_{y \in \sigma^{-1}(x)} u^a(y).
     \]
     Вектор выигрышей в вершине `x` переопределяется как `u(x) = u(\hat{p}^a(x))`.
   - Если `x ∈ X⁰` (ход случая), выигрыш в `x` определяется как математическое ожидание:
     \[
     u(x) = \sum_{y \in \sigma^{-1}(x)} p(y|x) \cdot u(y).
     \]

2. Получаем редуцированную игру, в которой финальными становятся вершины `Z₁`. Следующее множество `Z₂` – вершины, все преемники которых принадлежат `T ∪ Z₁`. Для них аналогично определяются оптимальные выборы и новые выигрыши.

3. Процесс продолжается, пока не достигнем начальной вершины `x₀`. Построенная ситуация `µ̂` и есть совершенное подыгровое равновесие.

**Теорема существования СПР:** В любой конечной позиционной игре с полной информацией существует совершенное подыгровое равновесие, определяемое алгоритмом Куна.

### 4. Позиционные игры общего вида (с неполной информацией)

В играх с неполной информацией игрок в момент хода может не различать некоторые позиции. Для каждого игрока `a` множество его личных позиций `Xᵃ` разбивается на **информационные множества** `Zⱼ, j ∈ Jᵃ`. Все позиции одного информационного множества имеют одинаковое число альтернатив. Игрок, делая ход, знает только информационное множество, но не конкретную позицию.

**Чистая стратегия** игрока `a` каждому информационному множеству `Zⱼ` предписывает одну альтернативу. Ситуация определяет вероятностное распределение на финальных вершинах и ожидаемые выигрыши.

### 5. Смешанные стратегии и стратегии поведения

- **Смешанная стратегия** `πᵃ` – вероятностное распределение на множестве чистых стратегий `{µᵃ}`. Рандомизация производится один раз перед началом игры.
- **Стратегия поведения** `βᵃ` – каждому информационному множеству `Zⱼ` сопоставляется вероятностное распределение на альтернативах: `p_{jk} ≥ 0`, `∑_k p_{jk} = 1`. Рандомизация происходит независимо в каждом множестве при каждом его посещении.

В общем случае смешанные стратегии и стратегии поведения **не эквивалентны**: с помощью смешанных стратегий можно достичь большего. Пример – антагонистическая игра с партнёром (пример 14.1 в лекциях): оптимальная смешанная стратегия даёт выигрыш 1/4, а любая стратегия поведения – только 0.

### 6. Игры с полной памятью

Игра называется **игрой с полной памятью** для игрока `a`, если для любой его чистой стратегии `µᵃ` и любого информационного множества `Zⱼ`, существенного для `µᵃ`, все позиции этого множества являются возможными для `µᵃ`. Содержательно игрок помнит все свои предыдущие ходы и не может забыть, каким путём он пришёл в информационное множество.

**Теорема Куна (1953):** В игре, имеющей полную память для всех игроков, смешанные стратегии и стратегии поведения эквивалентны в том смысле, что для любой смешанной стратегии найдётся стратегия поведения, приводящая к тем же распределениям на финальных вершинах при любых стратегиях партнёров. Следовательно, при поиске равновесия можно ограничиться стратегиями поведения, что радикально сокращает размерность задачи.

**Следствие для игр с полной информацией:** Поскольку в таких играх каждое информационное множество содержит ровно одну вершину, память всегда полная, и любое равновесие может быть реализовано в стратегиях поведения (в частности, через обратную индукцию алгоритмом Куна).

## 20 билет: Задачи распределения ресурсов. Теорема Гермейера. Дискретная задача распределения ресурсов. Лемма Гиббса.

### 1. Теорема Гермейера (иерархическая игра двух лиц)

Рассматривается игра, в которой первый игрок (центр) выбирает **стратегию‑функцию** `f: Y → X` и сообщает её второму. Второй, зная `f`, выбирает своё действие `y ∈ Y`, максимизируя свой выигрыш `G(x, y)`. Выигрыш первого есть `F(x, y)`. Первый игрок стремится к максимальному **гарантированному** выигрышу.

**Предположения:** `X, Y` – компактные множества, функции `F, G` непрерывны.

**Введём обозначения:**
- `G₂ = max_{y∈Y} min_{x∈X} G(x, y)` – максимальный гарантированный выигрыш второго игрока.
- `E = { y ∈ Y | min_{x∈X} G(x, y) = G₂ }` – множество «рациональных» выборов второго (гарантирующих ему этот максимин).
- `D = { (x, y) | G(x, y) > G₂ }` – пары, дающие второму строго больше его гарантированного выигрыша.
- `K = sup_{(x,y)∈D} F(x, y)` (если `D = ∅`, то `K = −∞`).
- `M = min_{y∈E} max_{x∈X} F(x, y)` – гарантированный выигрыш первого на множестве `E`.

**Теорема Гермейера:** Наилучший гарантированный выигрыш первого игрока равен
\[
F_2 = \max[K, M].
\]

**Интерпретация и построение оптимальной стратегии:**
- Если `K > M`, то для любого `ε > 0` первый может обеспечить себе выигрыш `K − ε` с помощью стратегии, которая «наказывает» второго при отклонении от `D` (грозит ему значением чуть выше `G₂`).
- Если `K ≤ M`, то оптимальна стратегия `f^M_0`: при `y ∈ E` выбирает `f^M_0(y) ∈ \arg\max_{x∈X} F(x, y)`; при `y ∉ E` выбирает `f^H(y)`, дающую второму выигрыш **меньше** `G₂`, вынуждая его оставаться в `E`. Тогда `W(f^M_0) = M`.

Таким образом, первый игрок либо «платит» второму больше его гарантированного выигрыша (если это выгодно), либо навязывает ему рациональные действия и получает `M`.

### 2. Задача оптимального распределения ресурсов (непрерывный случай)

Имеется `n` пунктов и ресурс объёмом `A > 0` (непрерывный). Функции `f_i(t)` – эффективность вложения ресурса в `i`-й пункт; они определены, непрерывны и возрастают на `[0, A]`, причём `f_1(0) ≤ f_2(0) ≤ … ≤ f_n(0)`. Распределение – вектор `x = (x_1, …, x_n)` с `x_i ≥ 0`, `∑ x_i = A` (множество `M₀`).

**Задача 1 (максимизация минимального эффекта):**
\[
\max_{x \in M_0} \min_{1 \le i \le n} f_i(x_i).
\]

**Принцип уравнивания (теорема):** Оптимальное распределение `x^0` существует и единственно. Для него найдётся номер `k` (`1 ≤ k ≤ n`) такой, что
\[
f_1(x_1^0) = \dots = f_k(x_k^0) < f_{k+1}(0), \quad x_i^0 = 0 \ \ (i > k).
\]
Если `f_1(0) = … = f_n(0)`, то `k = n`. Иными словами, ресурс распределяется между `k` «слабейшими» пунктами так, чтобы их эффекты сравнялись, а остальные пункты не получают ничего, так как их начальная отдача выше достигнутого уравненного уровня.

**Алгоритм:** решается система
\[
f_i(x_i) = C,\ i = 1..k,\quad \sum_{i=1}^k x_i = A,
\]
подбирая максимальное `k`, при котором `C < f_{k+1}(0)`.

### 3. Дискретная задача распределения ресурсов

Ресурс `A` – целое число, распределение `x_i ∈ ℕ₀`, `∑ x_i = A` (множество `M₀`). Функции `f_i(t)` – возрастающие по `t`. Задача:
\[
\max_{x \in M_0} \min_{1 \le i \le n} f_i(x_i).
\]

**Теорема (о значении эффекта):** Пусть `x^*` – оптимальное распределение с минимальным числом индексов, на которых достигается минимум. Тогда для любого `j` с `x_j^* > 0`:
\[
f_j(x_j^* - 1) \le \min_{1 \le i \le n} f_i(x_i^*) \le f_j(x_j^*).
\]
Иными словами, отнятие единицы ресурса у любого задействованного пункта не может сделать его эффект выше текущего минимума – иначе можно было бы перераспределить и улучшить общий минимум. Это даёт критерий оптимальности.

**Алгоритм решения:** стартуя с приближения (например, округлённого непрерывного решения), итеративно улучшают минимум, забирая единицу у пункта с наибольшим запасом над минимумом и передавая её другому, пока условие теоремы не выполнится.

### 4. Лемма Гиббса (задача максимизации суммарной эффективности)

Рассматривается задача оптимального распределения непрерывного ресурса с целью **максимизации суммарного эффекта**:
\[
\max_{x \in M_0} \sum_{i=1}^n f_i(x_i). \quad (II)
\]
Функции `f_i(t)` дифференцируемы (обычно вогнуты).

**Лемма Гиббса (Теорема 3.10):** Если `x^0` – оптимальное распределение в задаче (II), то существует такое `λ`, что
\[
\begin{cases}
f_i'(x_i^0) = \lambda, & \text{если } x_i^0 > 0, \\
f_i'(0) \le \lambda, & \text{если } x_i^0 = 0.
\end{cases}
\]
Если функции вогнуты, это условие является и достаточным. Кроме того, если производные упорядочены:
\[
f_1'(0) \ge f_2'(0) \ge \dots \ge f_n'(0) \quad \text{и} \quad f_i''(t) < 0,
\]
то существует `l` такое, что `x_i^0 > 0` для `i = 1..l` и `x_i^0 = 0` для `i > l`. Таким образом, ресурс распределяется в первую очередь в наиболее «прибыльные» пункты, пока уравниваются нормы прибыли (производные).

**Экономическая интерпретация:** `λ` – «теневая цена» ресурса. Предельная полезность вложения в каждый используемый пункт одинакова и равна `λ`, а в неиспользуемые пункты вложение невыгодно, так как начальная предельная полезность ниже `λ`.

## 21 билет: Средние и эмпирические операционные характеристики стратегий распознавания (классификаторов, регрессий). Проблема недообучения и переобучения. Проблема устойчивости решений. Роль обучающей, валидационной и контрольной выборок при построении распознающей системы. Скользящий контроль (кросс-валидация). Регуляризация на примере линейной регрессии, её роль.

### 1. Операционные характеристики (метрики качества)

Для оценки качества моделей распознавания используются различные метрики, основанные на сравнении предсказаний с истинными значениями.

#### 1.1. Для регрессий (по лекциям)

Пусть `y_i` – истинное значение, `ŷ_i` – предсказанное, `n` – объём выборки.

- **MAE (Mean Absolute Error)** – средний модуль отклонения:
  \[
  MAE = \frac{1}{n}\sum_{i=1}^{n} |y_i - \hat{y}_i|
  \]
- **MSE (Mean Squared Error)** – среднеквадратическая ошибка:
  \[
  MSE = \frac{1}{n}\sum_{i=1}^{n} (y_i - \hat{y}_i)^2
  \]
- **RMSE (Root Mean Squared Error)** – корень из MSE:
  \[
  RMSE = \sqrt{\frac{1}{n}\sum_{i=1}^{n} (y_i - \hat{y}_i)^2}
  \]
- **MAPE (Mean Absolute Percentage Error)** – средняя абсолютная процентная ошибка:
  \[
  MAPE = \frac{100\%}{n}\sum_{i=1}^{n} \left| \frac{y_i - \hat{y}_i}{y_i} \right|
  \]

#### 1.2. Для классификации (бинарной)

Вводятся понятия: TP (true positive), TN (true negative), FP (false positive), FN (false negative).

- **Accuracy** – доля правильных ответов:
  \[
  Acc = \frac{TP + TN}{TP + TN + FP + FN}
  \]
- **Precision** – точность (доля правильно предсказанных положительных среди всех предсказанных положительных):
  \[
  P = \frac{TP}{TP + FP}
  \]
- **Recall** – полнота (доля правильно предсказанных положительных среди всех истинно положительных):
  \[
  R = \frac{TP}{TP + FN}
  \]
- **F1‑мера** – среднее гармоническое точности и полноты:
  \[
  F_1 = 2 \cdot \frac{P \cdot R}{P + R}
  \]
- **Fβ‑мера** – обобщённая (β > 0, задаёт вес полноты относительно точности):
  \[
  F_\beta = (1 + \beta^2) \frac{P \cdot R}{\beta^2 P + R}
  \]
- **MCC (Matthews Correlation Coefficient)** – коэффициент корреляции Мэттьюса:
  \[
  MCC = \frac{TP \cdot TN - FP \cdot FN}{\sqrt{(TP+FP)(TP+FN)(TN+FP)(TN+FN)}} \in [-1, 1]
  \]
  Значение +1 – идеальное предсказание, 0 – случайное, -1 – полное несогласие.

### 2. Недообучение и переобучение

- **Недообучение (underfitting)** – алгоритм не обеспечивает достаточно малой ошибки на обучающей выборке. Возникает при использовании недостаточно сложных моделей (например, линейная регрессия для нелинейной зависимости).
- **Переобучение (overfitting)** – ошибка на обучающей выборке мала, но на тестовой выборке существенно выше. Возникает при использовании избыточно сложных моделей, которые «запоминают» шум и частные особенности обучающих данных, а не общие закономерности.

### 3. Проблема устойчивости решений

Под устойчивыми обучающими алгоритмами понимаются такие, которые дают решение, незначительно изменяющееся при малом изменении обучающей выборки. Неустойчивость проявляется, когда один и тот же алгоритм, запущенный на разных подвыборках (или с разными начальными условиями), приводит к сильно различающимся моделям. Это характерно для деревьев решений, нейронных сетей (из‑за случайной инициализации) и некоторых методов кластеризации. Устойчивость обычно повышают с помощью регуляризации, ансамблей или усреднения.

### 4. Обучающая, валидационная и контрольная выборки

- **Обучающая выборка (training set)** – данные, на которых производится настройка параметров модели.
- **Тестовая (контрольная) выборка (test set)** – независимые данные, не участвовавшие в обучении. По ним вычисляется несмещённая оценка качества построенной модели. Если эта оценка используется для выбора лучшей модели, она снова становится смещённой (оптимистичной).
- **Проверочная (валидационная) выборка (validation set)** – данные, по которым осуществляется выбор наилучшей модели из множества моделей, построенных на обучающей выборке. После выбора модели её качество оценивается на тестовой выборке, которая не использовалась ни для обучения, ни для выбора модели.

### 5. Скользящий контроль (кросс‑валидация)

**Кросс‑валидация (cross‑validation, CV)** – процедура эмпирического оценивания обобщающей способности алгоритмов. Фиксируется несколько разбиений исходной выборки на обучающую и контрольную подвыборки. Для каждого разбиения настраивается модель по обучающей части, затем вычисляется ошибка на контрольной части. **Оценкой скользящего контроля** называется средняя по всем разбиениям величина ошибки на контрольных подвыборках.

Если выборка независима, средняя ошибка скользящего контроля даёт **несмещённую** оценку вероятности ошибки. Это выгодно отличает её от ошибки на обучающей выборке, которая может быть оптимистически занижена (переобучение). Наиболее распространённый вариант – **k‑блочная кросс‑валидация** (k‑fold CV): выборка случайно разбивается на `k` блоков примерно равного размера; в каждом из `k` запусков один блок служит контрольной выборкой, остальные `k‑1` – обучающей.

### 6. Регуляризация на примере линейной регрессии

**Регуляризация** – метод добавления дополнительных ограничений (штрафа за сложность модели) к минимизируемому функционалу для предотвращения переобучения и решения некорректно поставленных задач.

**Линейная регрессия:** модель имеет вид `F_β(x) = ∑_{i=1}^n β_i f_i(x)`, где `f_i(x)` – признаки. Функционал эмпирического риска (MSE):
\[
Q(β) = \|F_β - y\|^2
\]

**L2‑регуляризация (гребневая регрессия, Ridge):** добавляется квадрат нормы коэффициентов:
\[
Q_λ(β) = \|F_β - y\|^2 + λ \sum_{i=1}^n β_i^2
\]
Параметр `λ ≥ 0` – гиперпараметр регуляризации. L2‑регуляризация штрафует большие веса, но не обнуляет их.

**L1‑регуляризация (лассо, Lasso):** добавляется сумма модулей коэффициентов:
\[
Q_λ(β) = \|F_β - y\|^2 + λ \sum_{i=1}^n |β_i|
\]
L1‑регуляризация приводит к разреженным решениям (многие веса становятся точно нулевыми), что полезно для отбора признаков.

**Роль регуляризации:**
- Снижает дисперсию оценок, уменьшая переобучение.
- Улучшает устойчивость решения (малые изменения данных не приводят к сильному изменению коэффициентов).
- В случае L1 – упрощает модель за счёт удаления незначимых признаков.
- Позволяет работать с некорректно поставленными задачами (например, когда количество признаков больше числа наблюдений).

## 22 билет: Проблема смещения-дисперсии. Ансамбли классификаторов. Основные этапы работы типичного базового классификатора, возможность коррекции на разных этапах. Бэггинг. Случайные подпространства. Бустинг. Случайный лес как композиция основных подходов к построению ансамбля.

### 1. Дилемма смещения‑дисперсии (bias‑variance tradeoff)

Ожидаемая ошибка обобщения (на новых данных) может быть разложена на три компоненты:
\[
\mathbb{E}[(y - \hat{f}(x))^2] = \text{Bias}[\hat{f}(x)]^2 + \text{Var}[\hat{f}(x)] + \sigma^2,
\]
где `σ²` – неустранимая ошибка (шум данных).

- **Смещение (bias)** – систематическая ошибка, возникающая из‑за слишком простой модели (недообучение). Модель не способна уловить истинную зависимость.
- **Дисперсия (variance)** – чувствительность модели к малым изменениям обучающей выборки. Высокая дисперсия характерна для сложных моделей (переобучение).

**Суть дилеммы:** нельзя одновременно минимизировать и смещение, и дисперсию. Увеличение сложности модели снижает смещение, но увеличивает дисперсию, и наоборот. Цель – найти баланс, минимизирующий суммарную ошибку.

### 2. Ансамбли классификаторов

**Ансамбль** – множество прогнозирующих алгоритмов, используемых для коллективного решения. Окончательное решение принимается, например, голосованием (возможно взвешенным). При определённых условиях ансамбли обладают точностью, значительно превосходящей точность отдельных классификаторов, и устойчивы к «зашумлению» обучающей выборки.

**Необходимое и достаточное условие высокой точности ансамбля:** составляющие его классификаторы должны быть:
- достаточно точны (ошибка меньше 0.5 для бинарной классификации);
- различны (совершают ошибки на разных прецедентах).

### 3. Основные этапы работы типичного базового классификатора

1. **Обучение** – по обучающей выборке строится модель (например, решающее дерево, нейронная сеть, метод опорных векторов).
2. **Классификация** – для нового объекта модель выдаёт предсказание (метку класса или вероятность).

**Возможность коррекции на разных этапах:**
- **Коррекция до обучения** – предобработка данных (масштабирование, отбор признаков, балансировка классов).
- **Коррекция во время обучения** – изменение весов объектов (бустинг), варьирование гиперпараметров, добавление регуляризации.
- **Коррекция после обучения** – комбинирование нескольких моделей (ансамбли), калибровка вероятностей.

### 4. Бэггинг (Bootstrap Aggregating)

**Идея:** уменьшить дисперсию путём построения нескольких независимых классификаторов на разных подвыборках, полученных с помощью бутстрепа (случайная выборка с возвращением того же размера, что и исходная выборка). Каждый классификатор обучается на своей бутстреп‑выборке. Итоговое предсказание – усреднение (для регрессии) или простое голосование (для классификации).

**Свойства:**
- Особенно эффективен для нестабильных алгоритмов с высокой дисперсией (решающие деревья, нейронные сети).
- Снижает дисперсию, помогает избежать переобучения.
- Классификаторы строятся параллельно (независимо друг от друга).

### 5. Случайные подпространства (Random Subspaces)

Метод, аналогичный бэггингу, но вместо выборки объектов случайным образом выбирается **подмножество признаков** (столбцов) для каждого базового классификатора. Каждый классификатор обучается на всех объектах, но только на выбранных признаках. Это увеличивает разнообразие членов ансамбля. Часто комбинируется с бэггингом.

### 6. Бустинг (Boosting)

**Идея:** последовательное наращивание ансамбля, где каждый следующий классификатор обучается на «трудных» примерах, на которых предыдущие ошибались. Веса объектов корректируются после каждой итерации: ошибки предыдущих классификаторов получают больший вес.

**Пример – AdaBoost (Adaptive Boosting):**
1. Инициализировать веса объектов равными `1/N`.
2. Для `t = 1..T`:
   - Обучить классификатор `h_t` на взвешенной выборке.
   - Вычислить взвешенную ошибку `ε_t`.
   - Вычислить вес классификатора `α_t = 0.5 * ln((1-ε_t)/ε_t)`.
   - Обновить веса объектов: ошибочно классифицированные увеличить, правильно – уменьшить; затем нормализовать.
3. Итоговый классификатор: `H(x) = sign(∑ α_t h_t(x))`.

**Свойства:**
- Снижает как смещение, так и дисперсию (особенно смещение).
- Классификаторы строятся последовательно, что затрудняет параллелизацию.
- Чувствителен к шуму (выбросы получают большой вес).

### 7. Случайный лес (Random Forest)

**Случайный лес** – ансамбль решающих деревьев, сочетающий в себе идеи бэггинга и случайных подпространств.

**Алгоритм построения случайного леса (для классификации/регрессии):**
- Для каждого дерева `b = 1..B`:
  - Сгенерировать бутстреп‑выборку из исходных данных (бэггинг).
  - Построить решающее дерево, но на каждом узле выбирать наилучший признак не из всех `p` признаков, а из случайного подмножества размера `m << p` (обычно `m ≈ √p` для классификации, `p/3` для регрессии). Это вносит дополнительную случайность и увеличивает разнообразие деревьев.
  - Дерево строится до полной глубины (без подрезания).
- Итоговое предсказание – голосование (классификация) или усреднение (регрессия).

**Преимущества:**
- Высокая точность, устойчивость к переобучению.
- Работает с большим числом признаков, может оценивать их важность.
- Хорошо параллелится (деревья строятся независимо).
- Не требует тщательной настройки гиперпараметров.

**Случайный лес как композиция подходов:**
- **Бэггинг** – бутстреп‑выборки и усреднение.
- **Случайные подпространства** – случайный выбор признаков в узлах.
- **Бустинг** – не используется (деревья строятся независимо, а не последовательно).

## 23 билет: Задачи кластеризации, сопоставление с операцией группирования и задачей классификации. Различные постановки: разбиение, стохастическая, нечёткая, иерархическая, упорядочивание, однокластерная (последовательная). Задача и процедура K-средних. Метод нечеткой кластеризации С-средних. Агломеративная кластеризация.

### 1. Определение кластеризации

**Кластеризация** – задача разбиения множества объектов на группы (кластеры) так, чтобы объекты внутри группы были похожими, а объекты разных групп – как можно более различными. Главное отличие от классификации: перечень групп (классов) заранее не задан и определяется в процессе работы алгоритма.

Этапы кластерного анализа:
1. Отбор выборки объектов.
2. Определение множества переменных (признаков) и нормализация.
3. Вычисление меры сходства (расстояния) между объектами.
4. Применение метода кластеризации.
5. Представление и анализ результатов.

### 2. Сопоставление с группированием и классификацией

- **Группирование** – разбиение на группы по заданному правилу (например, по значениям атрибута). Кластеризация же ищет естественные группы на основе сходства.
- **Классификация** – относится к обучению с учителем: известны метки классов для обучающих объектов. Кластеризация – обучение без учителя (метки неизвестны).

### 3. Различные постановки задачи кластеризации

- **Разбиение (partitional clustering)** – требуется разбить множество объектов на заданное число `k` непересекающихся кластеров. Пример: `k`‑средних.
- **Стохастическая кластеризация** – предполагает, что кластеры описываются вероятностными распределениями (смесь распределений). Пример: EM‑алгоритм.
- **Нечёткая кластеризация (fuzzy clustering)** – каждый объект принадлежит каждому кластеру с некоторой степенью (вероятностью). Степени принадлежности – вещественные числа от 0 до 1, сумма по кластерам для одного объекта равна 1. Пример: Fuzzy C‑means.
- **Иерархическая кластеризация** – строится древовидная структура вложенных кластеров (дендрограмма). Не требует априорного знания числа кластеров. Делится на агломеративные (снизу вверх) и дивизимные (сверху вниз).
- **Упорядочивание** – кластеры упорядочены (например, по времени или по рангу).
- **Однокластерная (последовательная)** – ищется один кластер (например, кластер наибольшей плотности), остальные объекты считаются шумом.

### 4. Задача и процедура K‑средних (K‑means)

**Назначение:** разбиение `n` объектов на `k` кластеров, минимизирующее сумму квадратов расстояний от каждого объекта до центра своего кластера.

**Алгоритм:**
1. **Инициализация** – выбрать `k` начальных центров кластеров (например, случайно или как наиболее удалённые друг от друга объекты).
2. **Распределение объектов** – каждый объект относится к ближайшему центру (по выбранной метрике, обычно евклидово расстояние).
3. **Пересчёт центров** – для каждого кластера вычислить новый центр как среднее арифметическое всех объектов, входящих в кластер.
4. **Повторять шаги 2‑3** до тех пор, пока центры не перестанут меняться (или изменения станут меньше порога).

**Формальное описание (по лекциям):**

1: сформировать начальное приближение центров кластеров `μ_y` (наиболее удалённые друг от друга объекты)
2: повторять
3:   отнести каждый объект к ближайшему центру: `y_i = arg min_{y} ρ(x_i, μ_y)`
4:   вычислить новые центры: `μ_{yj} = (∑_{i: y_i=y} f_j(x_i)) / (∑_{i: y_i=y} 1)`
5: пока `y_i` не перестанут изменяться

**Достоинства:**
- сравнительно высокая эффективность при простоте реализации;
- высокое качество кластеризации при правильном выборе `k`;
- возможность распараллеливания (распределение объектов по процессорам);
- множество модификаций.

**Недостатки:**
- количество кластеров `k` – параметр алгоритма;
- результат сильно зависит от инициализации центров;
- выбросы влияют на расчёт центров;
- сходимость к локальному оптимуму, а не глобальному;
- алгоритм неприменим к данным, для которых не определено понятие «среднего» (категориальные данные).

### 5. Метод нечеткой кластеризации C‑средних (Fuzzy C‑means, FCM)

**Назначение:** получение нечёткого разбиения, где каждый объект принадлежит всем кластерам с разной степенью.

**Входные данные:** множество объектов `x_i`, количество кластеров `K`, параметр размытости `m > 1` (обычно `m = 2`), матрица принадлежностей `U` размера `N×K` (элементы `u_{ik} ∈ [0,1]`, сумма по `k` равна 1).

**Алгоритм:**
1. Выбрать начальную матрицу принадлежностей `U` (например, случайно).
2. Вычислить центры кластеров:
   \[
   c_k = \frac{\sum_{i=1}^N u_{ik}^m x_i}{\sum_{i=1}^N u_{ik}^m}
   \]
3. Обновить матрицу принадлежностей:
   \[
   u_{ik} = \frac{1}{\sum_{j=1}^K \left( \frac{\|x_i - c_k\|}{\|x_i - c_j\|} \right)^{\frac{2}{m-1}}}
   \]
4. Вычислить критерий нечёткой ошибки:
   \[
   E^2(X,U) = \sum_{i=1}^N \sum_{k=1}^K u_{ik}^m \|x_i - c_k\|^2
   \]
5. Повторять шаги 2‑4, пока изменения `U` не станут меньше заданного порога.

**Ограничения:**
- Метод предполагает, что кластеры имеют сферическую форму и примерно одинаковую дисперсию по всем размерностям. Если кластеры имеют эллиптическую форму или различную дисперсию, FCM даёт плохие результаты. Для таких случаев применяют смеси Гауссиан (GMM) или алгоритмы на основе ковариационных матриц.

### 6. Агломеративная кластеризация (Agglomerative clustering)

**Иерархический подход снизу вверх:** на начальном шаге каждый объект образует отдельный кластер. Затем на каждой итерации два **ближайших** (наиболее похожих) кластера объединяются в один. Процесс продолжается до тех пор, пока не получится один кластер, содержащий все объекты. Результат – дендрограмма, по которой можно выбрать желаемое число кластеров.

**Меры расстояния между кластерами (linkage criteria):**
- **Ближний сосед (single linkage)** – минимальное расстояние между любыми объектами из разных кластеров. Обладает **цепочечным эффектом**: кластеры могут вытягиваться в длинные цепочки. Хорош для выделения кластеров ленточной формы.
- **Дальний сосед (complete linkage)** – максимальное расстояние между объектами разных кластеров. Цепочечного эффекта нет, но на ранних этапах может объединять довольно несхожие группы.
- **Средний сосед (average linkage)** – среднее расстояние между всеми парами объектов.
- **Метод Уорда (Ward)** – минимизация увеличения внутрикластерной дисперсии.

**Алгоритм Ланса‑Уильямса (обобщённая формула для пересчёта расстояний):**

```
Алгоритм 1.5 Агломеративная кластеризация Ланса‑Уильямса
1: инициализировать множество кластеров C₁ = { {x₁}, …, {xₗ} }, t = 1
2: для всех t = 2..ℓ:
3:   найти в C_{t‑1} два ближайших кластера (U,V) = arg min R(U,V); R_t = R(U,V)
4:   изъять U, V, добавить слитый кластер W = U ∪ V: C_t = (C_{t‑1} \ {U,V}) ∪ {W}
5:   для всех S ∈ C_t вычислить расстояние R(W,S) по формуле Ланса‑Уильямса
```

Формула Ланса‑Уильямса обобщает различные методы:
\[
R(W,S) = α_U R(U,S) + α_V R(V,S) + β R(U,V) + γ |R(U,S) - R(V,S)|
\]
где коэффициенты зависят от выбранного метода (single, complete, average, Ward).

**Свойства агломеративной кластеризации:**
- Не требует задания числа кластеров заранее.
- Результат можно визуализировать в виде дендрограммы.
- Вычислительная сложность – от `O(n²)` до `O(n² log n)`, что ограничивает применение на очень больших выборках.
- Порядок объединения детерминирован (если нет связей).
